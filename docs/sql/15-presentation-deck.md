## 자주 쓰는 용어를 쉬운 말로

발표 중에는 아래처럼 풀어서 말하면 듣는 사람이 따라오기 쉽다.

| 문서 용어 | 쉬운 말 |
|------|------|
| Row Lock | 같은 row를 다른 스레드가 함부로 못 건드리게 막는 락 |
| Range Lock | 특정 id 범위를 한꺼번에 보호하는 락 |
| Page Latch | 페이지 메모리가 깨지지 않게 잠깐 거는 락 |
| Pager Mutex | 캐시 관리 정보만 보호하는 락 |
| Header Lock | `next_id`, `row_count` 같은 작은 전역값을 지키는 락 |
| Head-of-Line Blocking | 앞에서 막혀서 뒤 요청이 같이 밀리는 현상 |
| thread-per-connection | 연결마다 스레드 하나를 붙이는 방식 |
| underflow | 노드 안의 key 개수가 너무 적어지는 문제 |
| race | 먼저 끝난 쪽 때문에 결과가 달라지는 문제 |

## 발표 전에 꼭 정정할 표현

현재 코드 기준으로 사실 관계를 먼저 맞춰두는 것이 좋다.

- `MAX_CONNECTIONS=128` 제한은 구현되어 있다.
- 연결마다 스레드 하나를 두는 `thread-per-connection` 전환도 구현되어 있다.
- keep-alive 에러 응답 정합성도 구현되어 있다.
- idle timeout은 현재 코드에는 아직 없다.
- 따라서 발표에서는 timeout을 "운영적으로 필요하다고 판단했고, 다음 서버 개선 포인트로 남아 있다"라고 말하는 편이 정확하다.
- `engine_lock`은 모든 write DML을 줄 세우는 락이 아니다.
- 현재 `engine_lock`의 핵심 역할은 `DDL은 혼자`, `DML은 같이` 들어가게 하는 입장 관리용 락이다.
- 실제 write 병목은 자주 부딪히는 row/page, `header_lock`, lock 충돌, B+ tree 위쪽 경합 쪽이 더 가깝다.

## Slide 3. 초기 서버 설계

### 화면에 띄울 내용

- 메인 스레드는 `accept()`만 담당
- 요청은 worker에게 넘겨 병렬 처리
- 처음에는 thread pool이 가장 자연스럽다고 판단

```text
초기 구상

┌───────────────┐
│  Main Thread  │  accept()
└───────┬───────┘
        │ submit(fd)
        ▼
┌─────────────────────────────┐
│         Thread Pool         │
│   worker1 worker2 worker3   │
│   worker4 ...               │
└──────────────┬──────────────┘
               │
               ▼
          SQL Engine
```

### 발표 멘트

서버 쪽은 처음에 메인 스레드가 연결만 받고, 실제 처리는 worker 스레드들이 병렬로 하는 구조를 생각했습니다.  
그래서 자연스럽게 thread pool로 설계를 시작했습니다.  
짧은 작업을 여러 개 나눠 처리하는 데에는 thread pool이 일반적으로 효율적이라고 생각했기 때문입니다.

그런데 그렇다고 SQL 전체를 매번 락해버리면 병렬 처리의 의미가 거의 사라집니다.  
그래서 읽기는 같이 통과시키고, 쓰기 쪽만 더 강하게 제어하는 구조를 먼저 떠올렸습니다.

## Slide 5. 왜 락을 여러 층으로 나눴는가

### 화면에 띄울 내용

```text
┌─────────────────────────────────────────────┐
│  Row / Range Lock                           │  논리 충돌 제어
├─────────────────────────────────────────────┤
│  Engine RWLock                              │  DDL 독점 / DML 공유
├─────────────────────────────────────────────┤
│  Page Latch                                 │  페이지 메모리 보호
├─────────────────────────────────────────────┤
│  Pager Mutex                                │  프레임 메타데이터 보호
├─────────────────────────────────────────────┤
│  Header Lock                                │  next_id, row_count 보호
└─────────────────────────────────────────────┘
```

### 발표 멘트

제가 중간에 확실히 느낀 건, 이걸 한 종류의 락으로 해결하면 안 된다는 점이었습니다.  
논리적으로 같은 row를 보호하는 문제와, 실제 페이지 메모리 바이트를 보호하는 문제, 그리고 캐시 메타데이터를 보호하는 문제는 서로 완전히 다릅니다.  
그래서 락을 한 번에 크게 거는 대신, 역할별로 계층을 나눠 설계했습니다.

## Slide 6. Row Lock / Range Lock

### 화면에 띄울 내용

- 위치: `db.c`, `lock_table.c`
- point lock: `WHERE id = X`
- range lock: `WHERE id > X`, `id <= X`
- 문장이 시작될 때 잡고 끝날 때 푼다
- 문장이 끝날 때까지 락을 유지하는 방식

```text
예시

SELECT * FROM users WHERE id = 10;
    → S lock on id=10

UPDATE users SET age = 30 WHERE id = 10;
    → X lock on id=10

SELECT * FROM users WHERE id > 5;
    → S range lock on [6, +INF]
```

### 발표 멘트

이 계층은 메모리 자체를 잠그는 락이 아니라, 데이터 의미를 지키기 위한 락입니다.  
실제로 OS가 row를 잠가주는 게 아니라, 메모리 안에 lock table을 두고 엔트리를 기록해두고, 충돌하면 `condvar`로 기다리게 만들었습니다.  
핵심은 "같은 row의 의미가 동시에 깨지지 않게 하자"는 것입니다.  
페이지가 달라도 결국 같은 `id=10`을 만지는 거면 여기서 막아야 합니다.

### 발표 포인트

- page latch를 풀었다고 row의 의미까지 안전한 건 아니다
- 데이터 의미를 지키는 락이 따로 필요하다
- 현재 한계는 `name='Alice'` 같은 `id` 외 조건에 대해 완전하게 막는 수준은 아니라는 점이다

## Slide 7. Engine RWLock

### 화면에 띄울 내용

- 위치: `db.c`
- `CREATE TABLE`, `DROP TABLE`만 `wrlock`
- 나머지 DML은 `rdlock`
- 역할: 스키마 변경을 혼자 들어가게 하는 입장 관리 락

```text
             Engine Lock

DDL  ─────────────────────────────► wrlock
DML  ─────────────────────────────► rdlock

즉,
"모든 write를 여기서 직렬화한다"가 아니라
"DDL은 혼자 들어가고 DML은 같이 들어간다"가 정확함
```

### 발표 멘트

이 부분은 발표에서 꼭 정확히 말해야 합니다.  
처음엔 저도 "엔진 락이 쓰기를 막는 거 아닌가?"라고 생각했는데, 현재 코드 기준으로는 write DML도 `rdlock`으로 들어갑니다.  
즉 이 락의 핵심 역할은 모든 write를 한 줄로 세우는 게 아니라, 스키마 변경 같은 DDL을 혼자 들어가게 만드는 입장 관리에 가깝습니다.

### 발표 포인트

- 실제 병목은 `engine_lock` 하나보다는 자주 부딪히는 row, 자주 쓰는 page, `header_lock`, B+ tree 위쪽 경합 쪽이 더 큼

## Slide 8. Page Latch / Pager Mutex / Header Lock

### 화면에 띄울 내용

```text
Page Latch
  - 페이지 내용 자체 보호
  - heap fetch/scan은 rlatch
  - insert/update/delete는 wlatch
  - B+ tree는 latch coupling + crab protocol

Pager Mutex
  - frame 관리 정보 보호
  - find_frame, pin_count, dirty, used_tick, hash index

Header Lock
  - 작은 전역 카운터 보호
  - next_id, row_count, last_heap_page_id
```

### 발표 멘트

여기서 제가 중요하게 본 건 "실제 데이터와 관리 정보를 같은 락으로 묶지 말자"는 것이었습니다.  
페이지 내용은 frame별 latch가 보호하고, 캐시 프레임의 메타데이터는 pager mutex가 짧게 보호합니다.  
그리고 `next_id`, `row_count` 같은 작은 전역값은 굳이 큰 락으로 묶지 않고 `header_lock`으로만 짧게 잡습니다.  
이렇게 나눠야 전체가 한 락에 물려서 직렬화되는 걸 피할 수 있습니다.

## Slide 9. 첫 번째 큰 문제: Head-of-Line Blocking

### 화면에 띄울 내용

```text
문제 상황: thread pool 4개 + keep-alive 16개

┌─────────────┐   fd1 (idle keep-alive)
│ worker 1    │─────────────────────────┐
├─────────────┤                          │
│ worker 2    │──────────────────── fd2 │
├─────────────┤                          │
│ worker 3    │──────────────────── fd3 │
├─────────────┤                          │
│ worker 4    │──────────────────── fd4 │
└─────────────┘                          │
                                         │
나머지 fd5~fd16  ────────────────► queue에서 계속 대기
```

### 발표 멘트

부하 테스트를 넣고 보니까 생각보다 느리고 요청 실패도 많았습니다.  
처음에는 락 문제인가 싶었는데, 실제 원인은 락이 아니라 처리 단위를 잘못 잡은 거였습니다.  
thread pool은 짧은 작업에는 좋은데, keep-alive 연결은 "요청 1건"이 아니라 "소켓 수명 전체"가 작업이 됩니다.  
결국 worker 하나가 `recv()` 대기에 오래 묶이면서, 뒤에 있는 요청이 처리조차 못 되고 밀리고 있었습니다.

### 인사이트

- 병목은 항상 lock contention이 아니다
- 처리 단위를 잘못 잡아도 시스템이 막힌다

## Slide 10. HoL Blocking 해결

### 화면에 띄울 내용

- thread pool 제거
- `accept()`마다 스레드 생성
- `pthread_detach`
- `MAX_CONNECTIONS=128`

```text
현재 구조

                accept()
                   │
                   ▼
┌───────────────────────────────┐
│         Main Thread           │
│  연결 수 확인 후 스레드 생성     │
└───────┬─────────┬─────────┬────┘
        │         │         │
        ▼         ▼         ▼
   conn thread conn thread conn thread
        │         │         │
        └──────► handle_client()
                     │
                     ▼
                 db_execute()
```

### 발표 멘트

그래서 여기서는 발상을 바꿨습니다.  
요청 단위가 아니라 연결 단위로 스레드를 붙이자고 생각했고, 결국 thread-per-connection 모델로 전환했습니다.  
이 방식은 메모리를 더 씁니다. 하지만 keep-alive 연결이 존재하는 상황에서는, 최소한 지금 구조에서는 이것보다 낫다고 판단했습니다.  
실제 MySQL 계열이 써온 방식과도 결이 맞아서 그 방향으로 갔습니다.

### 정확성 메모

- 현재 코드에는 `MAX_CONNECTIONS` 제한이 구현되어 있다
- idle timeout은 아직 없다
- 발표에서는 timeout을 "다음 서버 개선"으로 말하는 편이 정확하다

## Slide 11. 두 번째 큰 문제: Page latch만으로는 부족했다

### 화면에 띄울 내용

```text
문제 시나리오

Thread A: rlatch(page 5) -> row(id=1) 읽음 -> rlatch 해제
Thread B:                     wlatch(page 5) -> row(id=1) 수정 -> wlatch 해제
Thread A: "나는 방금 읽은 값이 아직도 유효하다고 생각함"

=> 물리 메모리는 안전했지만, 논리 의미는 이미 달라짐
```

### 발표 멘트

처음에는 B+ tree 탐색이나 heap 페이지 접근에 대한 동시성을 해결했으니까 큰 문제는 없을 거라고 생각했습니다.  
그런데 페이지 접근 동시성을 해결한 것과, 트랜잭션 의미를 지키는 건 전혀 다른 문제였습니다.  
A가 값을 읽고 페이지 락을 푼 직후 B가 같은 row를 바꿔버리면, A는 물리적으로는 안전하게 읽었지만 논리적으로는 이미 틀린 값을 기반으로 움직이게 됩니다.  
그 순간 "아, 이건 페이지 락 말고도 별도의 논리 락이 필요하다"는 걸 확실히 느꼈습니다.

### 인사이트

- 메모리 안전성과 데이터 의미 보호는 다르다
- page latch만으로 SQL 뜻까지 지킬 수는 없다

## Slide 12. scan 기반 UPDATE/DELETE race

### 화면에 띄울 내용

```text
기존 2-pass 구조

1차 scan
  heap_scan() -> 조건 일치 row의 id 수집

시간차 발생

2차 write
  모아둔 id를 실제로 update/delete

문제:
  1차에서 본 대상과
  2차에서 수정하는 대상 사이에
  다른 스레드가 끼어들 수 있음
```

### 발표 멘트

이건 발표에서 꼭 이야기하면 좋은 포인트입니다.  
`UPDATE`나 `DELETE`의 scan 경로는 처음에 "대상 찾기"와 "실제 수정"이 분리된 2-pass 구조였습니다.  
그런데 이렇게 되면 첫 번째 pass에서 본 row와, 두 번째 pass에서 실제로 고치는 row 사이에 시간차가 생깁니다.  
스캔이 끝나고 나면 latch는 풀려 있으니까, 그 사이 다른 스레드가 값을 바꾸거나 지워버릴 수 있습니다.

### 해결

- scan 결과에서 `id`를 먼저 수집
- 그 `id` 전체를 정렬 후 row X lock 획득
- 실제로 쓰기 전에 조건을 다시 확인

### 인사이트

- scan 결과를 메모리에 담아놨다고 안전한 게 아니다
- 그 목록이 바뀌지 못하게 먼저 막아야 한다

## Slide 13. 구현 중 만난 다른 동시성 버그들

### 화면에 띄울 내용

```text
1) keep-alive 에러 응답 불일치
   - 성공은 keep-alive, 에러는 close
   - 프로토콜 상태가 어긋남

2) ORDER BY 전역 상태 race
   - 정렬 비교 컨텍스트가 전역이면
     동시 요청끼리 서로 덮어쓸 수 있음

3) B+ tree internal key 수 부족 문제 미복구
   - leaf만 고치면 끝이 아니고
     부모까지 규칙이 맞게 복구해야 함
```

### 발표 멘트

구현하면서 느낀 건 동시성 버그가 꼭 큰 락 설계에서만 생기지 않는다는 점이었습니다.  
예를 들어 keep-alive 에러 응답 문제는 락 문제가 아니라 통신 상태가 어긋나는 문제였고,  
`ORDER BY`는 정렬 기준을 전역 변수에 두면 서로 덮어쓸 수 있었습니다.  
또 B+ tree delete는 leaf만 잘 지운다고 끝나는 게 아니라, 부모 internal node까지 규칙이 맞게 복구해야 진짜 끝이었습니다.

### 인사이트

- 전역 상태 하나도 동시성 버그의 시작점이 될 수 있다
- 데이터 구조 복구는 문제가 난 자리만 고쳐서는 안 된다

## Slide 14. 성능 병목 최적화

### 화면에 띄울 내용

- `find_frame()` 선형 탐색 제거
- `lock_release_all()` 전체 순회 제거
- `heap_scan()`이 rlatch를 오래 쥐는 문제 줄이기

```text
Before

find_frame(page_id)        -> O(n)
lock_release_all()         -> 전체 lock table 순회
heap_scan()                -> callback 끝날 때까지 rlatch 유지

After

page_id -> frame_idx hash  -> 평균 O(1)
owner list 기반 해제       -> 내가 가진 lock만 해제
page snapshot 후 해제      -> latch를 짧게만 보유
```

### 발표 멘트

이 부분에서 얻은 인사이트는 하나였습니다.  
"짧게 잡는 구간이라도 너무 자주 호출되면 병목이 된다"는 점입니다.  
한 번 보면 별거 아닌 연산도, 페이지 접근마다 반복되고 statement마다 반복되면 체감 병목이 됩니다.  
그래서 빅오보다 "얼마나 자주 지나가는 길인가"가 더 중요하다는 걸 꽤 크게 느꼈습니다.

## Slide 15. 아직 해결하지 못한 문제

### 화면에 띄울 내용

- writer starvation
- `id`가 아닌 조건을 완전하게 막는 기능 부족
- idle timeout 미구현

```text
writer starvation 예시

S S S S S S S S S S ...
        ^
        X writer가 여기서 기다리기 시작

그런데 뒤에서 새로운 S가 계속 들어오면
writer가 오래 밀릴 수 있음
```

### 발표 멘트

해결 못한 문제도 분명히 있습니다.  
대표적인 게 writer starvation, 즉 쓰기 작업이 계속 밀리는 문제입니다.  
지금 구조에서는 읽기 락이 계속 들어오는 상황에서, 쓰기 요청 하나가 오래 밀릴 수 있습니다.  
즉 읽기에는 유리한 구조이긴 한데, 쓰기 차례를 공평하게 보장하는 부분은 아직 부족합니다.  
이건 발표에서 숨기기보다, 제가 다음 단계로 어디를 보려는지 보여주는 포인트로 쓰는 게 오히려 좋다고 생각합니다.

### 인사이트

- 읽기 병렬성을 높이는 설계는 종종 쓰기 차례 보장과 충돌한다
- 동시성 설계는 "되냐 안 되냐"보다 "누가 얼마나 기다리느냐"의 문제이기도 하다

## Slide 16. 정리

### 화면에 띄울 내용

```text
제가 이 프로젝트에서 얻은 핵심 정리

1. 락은 하나로 해결하는 게 아니라 역할별로 나눠야 한다
2. 데이터 의미를 지키는 것과 메모리를 지키는 것은 다르다
3. 병목은 lock 자체보다 scheduling에서 시작될 수도 있다
4. 짧게 잡는 구간도 자주 지나가는 길이면 병목이 된다
5. 동시성은 "정답 하나"보다 서로 충돌하는 선택을 관리하는 문제다
```

### 발표 멘트

정리하면, 저는 이 프로젝트를 하면서 "동시성은 락 하나 잘 거는 문제"가 아니라는 걸 배웠습니다.  
논리 락, 페이지 락, 캐시 메타데이터 락, 전역 카운터 락이 각각 필요한 이유가 다 있었고,  
또 성능 문제도 단순히 CPU가 느린 게 아니라 처리 단위나 자주 지나가는 경로 설계에서 시작됐습니다.  
그래서 이 프로젝트는 SQL 엔진 구현이기도 했지만, 동시에 어디를 나눠 잠그고 어디를 통과시킬지 계속 판단하는 과정이었다고 말씀드리고 싶습니다.

## 3분 버전 발표 오프닝

안녕하세요. 저는 이번에 디스크 기반 SQL 엔진을 만들고, 그 위에 HTTP API 서버까지 붙여서 여러 요청을 동시에 처리하는 MiniDB를 구현했습니다.  
처음에는 메인 스레드가 연결만 받고 worker들이 병렬 처리하는 thread pool 구조를 생각했습니다. 그런데 부하 테스트를 해보니, keep-alive 연결이 worker를 오래 점유하면서 오히려 요청이 밀리는 Head-of-Line Blocking이 발생했습니다.  
그래서 서버 구조를 thread-per-connection으로 바꾸고, 연결 수를 제한하는 방식으로 구조를 재설계했습니다.

그다음 핵심은 동시성 계층을 나누는 일이었습니다.  
저는 이 문제를 한 종류의 락으로 해결하지 않고, row/range lock으로 데이터 의미 충돌을 막고, page latch로 페이지 메모리를 보호하고, pager mutex로 캐시 관리 정보를 보호하고, header lock으로 `next_id` 같은 작은 전역값을 보호하는 식으로 나눴습니다.  
이 과정에서 scan 기반 `UPDATE/DELETE` race, `ORDER BY` 전역 상태 race, B+ tree internal key 수 부족 문제 같은 버그를 발견하고 수정했습니다.

결국 이 프로젝트를 하면서 가장 크게 배운 건, 동시성은 단순히 "락을 건다"가 아니라 "무엇을 어떤 단위로 보호할지 나누는 문제"라는 점이었습니다.

## 7분 버전 발표 마무리 멘트

이 프로젝트는 작은 DB 엔진이지만, 실제로 구현하면서 느낀 문제는 굉장히 현실적이었습니다.  
keep-alive 때문에 thread pool이 막히기도 했고, 페이지 락만으로는 논리 정합성을 지킬 수 없었고, 전역 상태 하나 때문에 정렬에서 race가 나기도 했습니다.  
그래서 저는 이 프로젝트를 통해 단순히 SQL 기능을 늘리는 것보다, 동시성과 성능을 어떻게 분리해서 설계할지를 더 많이 배웠습니다.

아직 writer starvation 같은 숙제는 남아 있지만, 반대로 말하면 다음 단계가 분명한 상태라고 생각합니다.  
앞으로는 쓰기 차례 보장, 더 강한 조건 기반 락, idle timeout 같은 운영 보강까지 이어가면서 더 실사용에 가까운 구조로 확장해보고 싶습니다.

## 상세 해설 1. 프로젝트를 왜 이런 방향으로 잡았는가

제가 처음부터 이 프로젝트를 "디스크 기반 SQL 엔진 + API 서버"로 같이 보려고 했던 이유는, SQL 기능만 구현하면 실제로 어디서 문제가 생기는지 끝까지 보기 어렵다고 생각했기 때문이다.

메모리 안에서만 동작하는 SQL 인터프리터는 기능 구현 자체는 가능하다.  
하지만 실제로는 다음 문제가 빠르게 드러나지 않는다.

- 디스크 페이지를 어떤 단위로 읽고 쓸 것인가
- 페이지를 캐시하면 어떤 메타데이터가 필요해지는가
- 여러 요청이 동시에 들어왔을 때 어디에서 충돌이 나는가
- keep-alive 같은 네트워크 특성이 서버 설계를 어떻게 바꾸는가
- 논리 정합성과 물리 정합성이 언제부터 분리되는가

그래서 저는 SQL 문법과 실행기만 만든 게 아니라, "실제로 요청이 들어와서 DB 파일까지 닿는 전체 경로"를 구현해보는 쪽으로 목표를 잡았다.

```text
내가 보고 싶었던 전체 경로

Client
  │
  ▼
HTTP request
  │
  ▼
Socket server
  │
  ▼
db_execute()
  │
  ▼
Planner / Executor
  │
  ▼
Pager / Heap / B+ Tree
  │
  ▼
Disk file
```

이렇게 해야 동시성 문제도 진짜처럼 보인다.  
왜냐하면 DB 내부 로직만 보면 안 보이던 문제가 서버 구조를 올리는 순간 드러나기 때문이다.

## 상세 해설 2. 초기 설계 당시의 생각 흐름

초기 설계 당시 제 생각 흐름은 대략 아래와 같았다.

### 1단계: 병렬 요청을 받으려면 메인 스레드는 가벼워야 한다

메인 스레드가 모든 요청을 직접 처리하면 결국 전체가 싱글 스레드처럼 굴러간다.  
그래서 메인 스레드는 `accept()`만 하고, 실제 일은 worker가 처리하도록 분리해야겠다고 생각했다.

### 2단계: worker를 고정 개수로 두는 게 깔끔해 보였다

처음에는 thread pool이 자연스럽게 보였다.

- worker 수를 제한할 수 있다
- 스레드 생성/소멸 비용을 줄일 수 있다
- 메인 스레드는 큐에 작업만 넣으면 된다

즉 "짧은 작업이 계속 들어오는 서버"를 상상하면 thread pool은 매우 타당해 보였다.

### 3단계: 병렬로 처리하려면 DB 전체를 잠그면 안 된다

이 지점에서 동시성 설계를 먼저 고민했다.

- 읽기끼리는 보통 충돌하지 않는다
- 쓰기와 읽기, 쓰기와 쓰기는 충돌할 수 있다
- 그런데 모든 SQL 요청마다 DB 전체에 mutex를 걸면 사실상 싱글 스레드와 다를 게 없다

그래서 처음 떠올린 방향이 "읽기는 같이 들어가고, 쓰기 쪽만 더 강하게 제어하자"는 것이었다.

### 4단계: 데이터는 페이지 단위로 움직이니 페이지 단위 병렬성도 가능하지 않을까

Pager가 전체 DB를 한 번에 메모리에 올리는 게 아니라 필요한 페이지를 가져오는 구조이므로,  
서로 다른 페이지를 읽거나 수정하는 작업은 병렬화 가능성이 있다고 봤다.

여기서 생각이 더 확장됐다.

- 페이지 내용은 page latch로 보호하자
- 캐시 프레임 상태는 pager mutex로 보호하자
- 작은 전역 카운터는 header lock으로 따로 보호하자

즉 저는 구현 전에 이미 "락은 한 종류로 해결하면 안 된다"는 감각을 어느 정도 갖고 시작했다.

## 상세 해설 3. 전체 아키텍처 설명

현재 구조를 발표에서 한 번에 보여줄 때는 아래 그림이 가장 설명력이 좋다.

```text
┌────────────────────────────────────────────────────────────────────┐
│                          Client Layer                              │
│        curl / browser / stress test / custom socket clients        │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│                         Server Layer                               │
│  Main thread: accept()                                             │
│  Connection thread: HTTP parse -> handle_client -> db_execute      │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│                      SQL Execution Layer                           │
│  parse -> plan -> execute                                          │
│  SELECT / INSERT / UPDATE / DELETE / DROP / EXPLAIN               │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│                    Concurrency Control Layer                       │
│  Row/Range Lock -> Engine RWLock -> Page Latch -> Pager Mutex      │
│  plus Header Lock                                                  │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│                        Storage Layer                               │
│  Heap Pages + B+ Tree Index + Pager Cache + Free Page List         │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│                           Disk File                                │
│                         single .db file                            │
└────────────────────────────────────────────────────────────────────┘
```

### 발표용 한 줄 요약

이 시스템은 "요청을 받는 서버"와 "SQL을 실행하는 엔진"과 "페이지를 다루는 저장 계층"이 따로 노는 것이 아니라, 한 요청이 이 모든 계층을 순서대로 관통한다.

## 상세 해설 4. 요청 하나가 처리되는 실제 흐름

발표 중간에 구체적인 예시로 `UPDATE users SET age = 30 WHERE id = 10` 하나를 따라가면 구조 설명이 훨씬 쉬워진다.

```text
1) 클라이언트가 HTTP 요청 전송
2) 서버가 요청을 읽고 SQL 문자열 추출
3) db_execute() 호출
4) parser가 statement 생성
5) 필요한 row/range lock 획득
6) engine lock 진입
7) planner가 access path 선택
8) executor가 B+ tree / heap 페이지 접근
9) page latch로 물리 접근 보호
10) pager가 frame 메타데이터 관리
11) 결과 반환
12) lock release
13) HTTP 응답 전송
```

이 과정을 발표에서 보여주면, 락이 왜 여러 개 필요한지도 자연스럽게 연결된다.  
요청 하나가 한 번에 "데이터 의미를 지키는 계층", "메모리를 지키는 계층", "캐시를 관리하는 계층", "통신 계층"을 모두 건드리기 때문이다.

## 상세 해설 5. 동시성 계층을 왜 이렇게 나눴는가

이 부분은 발표의 핵심이다.  
저는 "락을 많이 넣었다"가 아니라 "같은 문제를 서로 다른 락으로 푸는 건 잘못이라고 판단했다"는 식으로 설명하는 편이 좋다고 본다.

### Row Lock / Range Lock

이 계층은 SQL의 논리 의미를 보호한다.

- `WHERE id = 10` 같은 단건은 point lock
- `WHERE id > 5` 같은 범위는 range lock
- statement 단위로 잡고 statement 끝에서 해제
- 문장이 끝날 때까지 락을 들고 가는 방식에 가깝다

이 락이 필요한 이유는 page latch로는 row의 의미를 끝까지 보호할 수 없기 때문이다.  
page latch는 "지금 이 메모리 바이트를 동시에 건드리지 마라"는 성격이고,  
row lock은 "내가 이 row를 기준으로 작업을 끝낼 때까지 다른 스레드가 의미를 바꾸지 마라"는 성격이다.

```text
Page latch가 지키는 것
  -> 페이지 메모리가 깨지지 않게 함

Row lock이 지키는 것
  -> SQL에서 보는 데이터 의미를 지킴
```

### Engine RWLock

이 락은 "누가 엔진 안으로 들어올 수 있는가"를 정하는 입장 관리 락이다.

- DDL은 혼자 들어가야 한다
- DML은 동시에 들어와도 된다

따라서 발표에서 "`engine_lock`이 모든 write를 막는다"라고 말하면 현재 코드 기준으로는 틀리다.  
정확한 설명은 "DDL은 독점, DML은 공유"다.

### Page Latch

이 락은 페이지 메모리 자체를 보호한다.

- heap page의 slot array
- 실제 row 데이터
- B+ tree leaf/internal node

특히 B+ tree는 단순한 lock/unlock이 아니라 latch coupling과 crab protocol을 사용한다.  
즉 부모를 잡은 상태에서 자식을 잡고, 안전하다고 판단되면 조상 래치를 빨리 풀어 병렬성을 확보한다.

### Pager Mutex

이 락은 frame 관리 정보용이다.

- `find_frame`
- `pin_count`
- `dirty`
- `used_tick`
- frame hash index

발표 포인트는 이거다.

> 페이지 내용과 페이지 캐시 관리 정보를 같은 락으로 보호하면, 시스템이 필요 이상으로 한 줄로 서게 된다.

그래서 저는 실제 데이터와 관리 정보를 분리했다.

### Header Lock

이 락은 아주 작지만 꼭 필요하다.

- `next_id`
- `row_count`
- `last_heap_page_id`

예를 들어 INSERT 두 개가 동시에 들어와도 같은 `next_id`를 가져가면 안 된다.  
또 새 heap page를 동시에 연결하다가 체인이 끊겨도 안 된다.  
이건 큰 락이 아니라 작은 전역 상태 보호 문제이므로, 별도 mutex로 짧게 처리하는 게 맞았다.

## 상세 해설 6. Head-of-Line Blocking을 어떻게 발견했는가

이 문제는 구현하다가 자연스럽게 나온 것이 아니라, 부하 테스트를 걸어보면서 체감으로 발견했다.

제가 본 현상은 대략 이랬다.

- 요청이 몰리면 처리량이 기대보다 많이 안 나온다
- 실패 요청도 꽤 나온다
- DB 내부 락이 문제라고 보기엔 CPU가 완전히 꽉 차지도 않는다

처음엔 row lock 충돌이나 page latch 때문인가 싶었다.  
그런데 흐름을 자세히 따라가 보니, 실제 병목은 SQL 엔진 안쪽이 아니라 서버 스케줄링 구조였다.

### 왜 thread pool이 여기서 안 맞았는가

thread pool은 "작업"의 단위가 짧을 때 잘 맞는다.  
하지만 keep-alive 연결이 있는 HTTP 서버에서는 작업의 단위가 요청 1건이 아니라 소켓 생명주기 전체로 늘어난다.

즉 worker는 이렇게 행동한다.

```text
worker 1
  -> fd 하나를 맡음
  -> 요청 1개 처리
  -> keep-alive 이므로 같은 fd에서 recv() 대기
  -> 이 동안 다른 연결은 못 받음
```

그러면 실제로는 CPU가 놀고 있어도, 스레드 슬롯이 점유되어 있기 때문에 뒤 요청이 못 들어온다.

### 이 문제에서 얻은 인사이트

- 동시성 문제는 lock table에서만 생기지 않는다
- thread pool이 항상 정답은 아니다
- 네트워크 프로토콜의 연결 유지 특성이 서버 스케줄링 모델을 바꿀 수 있다

이 부분은 발표 때 굉장히 좋은 포인트다.  
"락을 잘못 걸어서 느려진 게 아니라, 아예 처리 단위를 잘못 잡았다"는 이야기이기 때문이다.

## 상세 해설 7. 왜 thread-per-connection으로 바꿨는가

thread-per-connection으로 바꾸면 분명 단점도 있다.

- 스레드 수가 많아지면 메모리를 더 쓴다
- 연결 수를 그대로 무한히 받으면 위험하다

그런데 현재 구조와 요구사항에서는 장점이 더 컸다.

- keep-alive 연결이 다른 연결 dispatch를 막지 않는다
- 메인 스레드는 계속 `accept()`에 집중할 수 있다
- 각 연결이 독립적인 실행 흐름을 가진다

즉 저는 "스레드 생성 비용"보다 "keep-alive가 worker를 독점하는 비용"이 더 크다고 판단했다.

```text
Before
  worker slot 고정
  -> keep-alive 소켓이 worker를 오래 점유

After
  connection마다 thread
  -> idle 연결은 자기 thread만 점유
  -> 다른 새 연결은 따로 진입 가능
```

### 여기서 발표 시 꼭 정확히 말할 점

현재 코드에는 다음이 구현되어 있다.

- `MAX_CONNECTIONS=128`
- `pthread_detach`
- graceful shutdown

현재 코드에는 아직 다음이 구현되어 있지 않다.

- idle timeout
- `max requests per connection`

즉 발표에서는 "timeout까지 이미 구현했다"라고 말하기보다,  
"실제로 운영한다면 idle timeout과 요청 수 제한을 추가하는 게 맞고, 현재는 다음 개선 포인트로 본다"라고 말하는 편이 정확하다.

## 상세 해설 8. page latch만으로 해결되지 않은 이유

이 부분은 많은 질문이 들어올 수 있으므로, 메모리 안전성과 데이터 의미 보호를 나눠서 설명하는 것이 좋다.

### 메모리 안전성

메모리 안전성은 "페이지 안의 구조가 깨지지 않는가"의 문제다.

- slot array가 중간 상태로 읽히면 안 된다
- free slot chain이 꼬이면 안 된다
- B+ tree node split/merge 중간 상태를 다른 스레드가 보면 안 된다

이건 page latch가 해결한다.

### 데이터 의미 보호

데이터 의미 보호는 "내가 읽은 row의 뜻이 작업이 끝날 때까지 유지되는가"의 문제다.

```text
Thread A
  row(id=1, balance=1000) 읽음
  -> 이 값을 기반으로 계산 시작

Thread B
  같은 row를 500으로 수정

Thread A
  여전히 1000을 기준으로 후속 작업 수행
```

여기서는 page latch가 아무리 완벽해도 부족하다.  
왜냐하면 A가 값을 읽고 latch를 푼 이후의 시간을 보호하지 못하기 때문이다.

그래서 여기서 필요한 게 row lock이다.

### 발표 포인트

> page latch는 메모리를 보호한다.  
> row lock은 의미를 보호한다.

이 한 문장으로 설명하면 전달력이 좋다.

## 상세 해설 9. scan 기반 UPDATE/DELETE race를 어떻게 고쳤는가

이 문제는 단순 버그 수정이 아니라, scan 기반 DML이 원래 어떤 위험을 갖는지 보여준다.

### 기존 구조

```text
1차 pass
  heap_scan()
  -> 조건 맞는 row의 id 수집

2차 pass
  수집한 id들을 실제로 update/delete
```

겉보기에는 합리적이다.  
스캔 도중 구조를 바꾸면 순회가 깨질 수 있으니, 먼저 모아두고 나중에 수정하는 방식이다.

### 하지만 실제로는 시간차가 생긴다

문제는 1차와 2차 사이에 다른 스레드가 끼어들 수 있다는 것이다.

- 같은 row를 먼저 수정할 수도 있다
- 이미 삭제해버릴 수도 있다
- 조건이 더 이상 맞지 않을 수도 있다

즉 "내가 모아둔 id 목록"은 그 순간부터 오래된 정보가 된다.

### 해결 방식

저는 해결을 이렇게 했다.

1. scan 결과로 `id`를 수집한다
2. 그 `id` 전체에 대해 X row lock을 잡는다
3. 실제로 쓰기 직전에 지금도 조건이 맞는지 다시 확인한다
4. 여전히 조건이 맞는 row만 수정/삭제한다

### 이 접근의 의미

이건 단순히 "락을 더 걸었다"가 아니다.  
실제로는 "scan으로 얻은 대상 목록을 논리적으로 봉인했다"는 의미다.

이 표현은 발표 때 꽤 좋다.  
메모리에 목록이 있다는 사실과, 그 목록이 여전히 유효하다는 사실은 다르다는 걸 잘 보여주기 때문이다.

## 상세 해설 10. ORDER BY 전역 상태 race

이 문제는 작은 상태 공유가 얼마나 위험한지 보여준다.

정렬은 보통 `qsort` 같은 라이브러리를 써서 돌리는데, 비교 함수가 어떤 컬럼으로 비교할지, 오름차순인지 내림차순인지 같은 컨텍스트를 알아야 한다.  
이걸 전역 변수로 두면 단일 스레드에서는 편하다.

하지만 멀티스레드에서는 바로 문제가 된다.

```text
Thread A
  ORDER BY age DESC 수행
  -> 전역 정렬 컨텍스트 = age, DESC

Thread B
  ORDER BY name ASC 수행
  -> 전역 정렬 컨텍스트 = name, ASC

Thread A
  아직 sort 중인데 비교 기준이 바뀜
```

즉 데이터 구조는 멀쩡해도 정렬 기준 자체를 공유하면 서로 덮어쓰는 문제가 난다.

### 이 문제의 발표 포인트

- 동시성 버그는 lock table, pointer, page access 같은 거대한 구조에서만 생기지 않는다
- 전역 비교 기준 같은 작은 상태에서도 생긴다

이런 사례를 하나 끼워 넣으면 "동시성을 꽤 넓게 봤구나"라는 인상을 줄 수 있다.

## 상세 해설 11. B+ tree internal key 수 부족 문제의 의미

삭제 로직은 발표에서 종종 간단히 넘어가지만, 실제로는 삽입보다 더 까다로운 경우가 많다.

leaf에서 key를 지웠다고 끝나는 게 아니다.

- leaf의 key 수가 너무 적어질 수 있다
- leaf를 borrow/merge하면 부모 separator key가 바뀔 수 있다
- 부모도 최소 개수보다 적어질 수 있다
- 결국 루트 높이 축소까지 전파될 수 있다

즉 삭제는 "한 노드 수정"이 아니라 "불변식 복구 연쇄"다.

### 왜 중요했는가

처음엔 root shrink 정도만 구현되어 있어도 동작 자체는 되는 것처럼 보일 수 있다.  
하지만 internal node의 key 수가 너무 적어진 상태를 그냥 두면,

- 트리 높이가 불필요하게 커진다
- separator key 정합성이 약해질 수 있다
- 장기적으로 구조가 비정상적인 형태로 굳는다

그래서 이건 즉시 crash가 나는 버그가 아니더라도 꼭 닫아야 하는 문제였다.

### 발표용 한 문장

> 데이터 구조 복구는 에러가 발생한 노드 하나만 고치는 게 아니라, 불변식이 닿는 경로 전체를 닫아야 끝난다.

## 상세 해설 12. 자주 지나가는 경로 최적화에서 얻은 교훈

이번 프로젝트에서 성능 최적화는 거창한 알고리즘 변경보다, "자주 반복되는 짧은 경로"를 줄이는 쪽이 더 효과적이었다.

### `find_frame()` 해시화

처음에는 frame 배열을 선형 탐색해도 큰 문제처럼 보이지 않는다.  
하지만 page 접근은 거의 모든 요청에서 반복된다.

즉,

- 한 번의 비용은 작다
- 하지만 거의 모든 요청 경로에서 반복된다
- 그래서 누적 병목이 된다

이 때문에 `page_id -> frame_idx` 해시 인덱스로 바꿨다.

### `lock_release_all()` owner-list화

statement가 끝날 때마다 전체 lock table을 순회해서 "내 lock이 뭔지" 찾는 건 비효율적이다.  
그래서 스레드별 보유 lock 목록을 유지하도록 바꿨다.

핵심은 이거다.

```text
Before
  전체를 뒤져서 내가 가진 lock을 찾음

After
  내가 가진 lock 목록을 직접 들고 있다가 바로 해제
```

### `heap_scan()` snapshot 후 latch 조기 해제

이건 특히 중요한 최적화였다.  
scan callback이 오래 걸리는 동안 page rlatch를 계속 들고 있으면 writer가 그 페이지에 못 들어온다.  
그래서 page 안의 live row를 snapshot으로 복사해두고, latch는 빨리 풀고, callback은 latch 없이 수행하도록 바꿨다.

### 여기서 얻은 교훈

> 빅오도 중요하지만, 진짜 병목은 자주 지나가는 경로에서 나오는 경우가 많다.

## 상세 해설 13. 현재 구조의 장점과 한계

현재 구조의 장점은 꽤 명확하다.

- 단일 `.db` 파일 기반 디스크 저장 구조를 실제로 구현했다
- Heap + B+ tree + Pager cache가 분리되어 있다
- HTTP API 서버까지 이어져 외부 요청 경로를 끝까지 볼 수 있다
- 동시성 계층을 데이터 의미 / 메모리 / 관리 정보 기준으로 나눴다
- 단순한 기능 구현이 아니라 실제 race와 병목을 발견하고 수정했다

하지만 한계도 분명하다.

- writer starvation, 즉 쓰기 작업이 오래 밀리는 문제가 남아 있다
- `id`가 아닌 조건을 완전하게 막는 수준은 아니다
- idle timeout이 아직 없다
- 다중 테이블과 2차 인덱스는 아직 없다
- WAL, recovery, MVCC는 범위 밖이다

즉 발표에서는 "완성된 상용 DB"처럼 말하는 것보다,  
"교육용 MiniDB이지만 실제 시스템 설계 문제를 끝까지 경험한 구현"이라고 말하는 편이 가장 정확하고 설득력 있다.

## 상세 해설 14. 발표용 추천 스토리텔링 방식

발표는 기능 나열보다 "생각 -> 구현 -> 문제 -> 수정 -> 배운 점" 구조가 훨씬 기억에 남는다.

추천 흐름은 아래와 같다.

### 도입

- 나는 디스크 기반 SQL 엔진에 API 서버까지 붙이고 싶었다
- 여러 요청을 동시에 처리하는 구조를 만들고 싶었다

### 첫 설계

- 메인 스레드는 accept만
- worker는 SQL 실행
- 그래서 thread pool을 먼저 선택했다

### 첫 번째 깨달음

- keep-alive가 worker를 오래 점유한다
- 병목은 락이 아니라 처리 단위였다
- 그래서 thread-per-connection으로 바꿨다

### 두 번째 깨달음

- 페이지 락만으로는 SQL 의미론이 안 지켜진다
- 그래서 row/range lock이 필요했다

### 세 번째 깨달음

- 동시성 버그는 큰 락뿐 아니라 전역 상태, scan 구조, 자료구조 복구에서도 생긴다

### 마무리

- 결국 이 프로젝트는 SQL 기능 구현만이 아니라
- "어떤 단위로 보호하고 어떤 단위로 병렬화할지 결정하는 과정"이었다

이 흐름은 청중이 구현 내용을 따라오기 쉽고, 각 문제의 필요성이 자연스럽게 전달된다.

## 예상 질문과 답변

### Q1. 왜 thread pool 대신 thread-per-connection을 선택했나요?

thread pool 자체가 나쁜 건 아닙니다.  
문제는 keep-alive 연결이 있는 HTTP 서버에서는 처리 단위가 요청 1건이 아니라 소켓이 살아 있는 시간 전체로 늘어난다는 점입니다.  
그래서 worker가 `recv()` 대기에 오래 점유되면서 뒤 요청이 밀렸고, 현재 구조에서는 connection 단위로 분리하는 쪽이 더 맞다고 판단했습니다.

### Q2. thread-per-connection이면 메모리를 많이 쓰지 않나요?

맞습니다. 그래서 `MAX_CONNECTIONS` 제한을 두었습니다.  
현재 구조는 "병목을 먼저 해결하고, 그다음 운영 정책으로 제한한다"는 방향입니다.  
다음 단계로는 idle timeout과 요청 수 제한을 넣는 것이 자연스럽습니다.

### Q3. engine lock이 있으면 결국 싱글 스레드 아닌가요?

아닙니다. 현재 코드에서 DML은 `rdlock`으로 동시 진입합니다.  
즉 engine lock은 주로 DDL만 혼자 들어가게 하기 위한 락입니다.  
실제 병목은 hot row/page, `header_lock`, B+ tree 상단 경합 같은 쪽이 더 큽니다.

### Q4. row lock이 있는데 왜 page latch도 필요한가요?

row lock은 논리 의미를 보호하고, page latch는 메모리 구조를 보호합니다.  
row lock만 있으면 slot 배열이나 B+ tree 노드가 중간 상태로 노출될 수 있고,  
page latch만 있으면 latch를 푼 뒤 row 의미가 바뀌는 걸 막지 못합니다.  
둘은 대체 관계가 아니라 역할이 다릅니다.

### Q5. writer starvation은 왜 생기나요?

현재 lock table은 보유 중인 lock 기준으로 충돌을 판단합니다.  
대기 중인 writer가 있어도 새로운 reader가 계속 들어올 수 있으므로, read-heavy 환경에서는 writer가 오래 밀릴 수 있습니다.

### Q6. 그럼 writer starvation은 어떻게 해결할 수 있나요?

writer-preference 정책이나 대기열 순서를 더 공평하게 보는 방식을 넣을 수 있습니다.  
예를 들어 writer가 대기 중이면 이후 새로운 reader 진입을 잠시 막는 방식이 대표적입니다.

### Q7. 왜 `id`가 아닌 조건은 완전하게 못 막나요?

현재 row/range lock은 `id` 기준으로 설계되어 있기 때문입니다.  
`name='Alice'`나 `age > 20` 같은 조건은 scan 후 개별 row를 잠그는 수준이라, 조건 전체를 완전히 막는 수준까지는 가지 못했습니다.

### Q8. keep-alive timeout은 구현되어 있나요?

현재 코드에는 구현되어 있지 않습니다.  
발표에서는 "운영적으로 필요하다고 보고 다음 서버 개선 포인트로 남겨둔 상태"라고 말하는 것이 정확합니다.

### Q9. 가장 기억에 남는 버그는 무엇인가요?

Head-of-Line Blocking과 scan 기반 DML race입니다.  
둘 다 처음에는 락 문제처럼 보였지만, 실제로는 하나는 스케줄링 단위 문제였고 다른 하나는 scan 구조의 시간차 문제였습니다.

### Q10. 이 프로젝트에서 가장 크게 배운 점은 무엇인가요?

동시성은 락 하나 잘 거는 문제가 아니라, "무엇을 어떤 단위로 보호할지 나누는 문제"라는 점입니다.  
데이터 의미를 지키는 일, 메모리를 지키는 일, 캐시 관리 정보를 지키는 일이 각각 따로 있다는 걸 체감했습니다.

## 발표자가 기억하면 좋은 핵심 문장 모음

아래 문장들은 발표 중간중간 넣으면 전달력이 좋다.

- "처음에는 락 문제라고 생각했는데, 실제로는 스케줄링 단위 문제였습니다."
- "페이지 락은 메모리를 보호하지만, row의 의미까지 보호하지는 못합니다."
- "row lock은 논리 정합성을, page latch는 물리 정합성을 지키는 역할입니다."
- "이 프로젝트는 SQL 기능 구현이기도 했지만, 동시에 보호 단위를 나누는 연습이기도 했습니다."
- "짧게 잡는 구간이라도 너무 자주 지나가면 병목이 됩니다."
- "데이터 구조 복구는 문제가 난 노드 하나만 고치면 끝나는 게 아니라, 불변식이 닿는 경로 전체를 닫아야 합니다."
- "읽기 병렬성을 높이면 쓰기 차례를 공평하게 주기 어려워지는 충돌이 생깁니다."

## 발표용 결론 문단

이번 프로젝트에서 저는 디스크 기반 SQL 엔진을 만드는 것 자체도 중요했지만, 그 엔진이 서버 위에서 여러 요청을 동시에 받을 때 어떤 문제가 생기는지를 끝까지 보는 게 더 중요하다고 생각했습니다.  
실제로 구현해보니 서버 스케줄링, 논리 락, 페이지 래치, 캐시 메타데이터 보호, 자료구조 복구, 프로토콜 정합성이 전부 연결되어 있었고, 어느 한 부분만 잘 만든다고 시스템이 안정적으로 동작하지는 않았습니다.

그래서 이 프로젝트는 단순히 SQL 기능을 추가한 구현이 아니라,  
"병렬성을 유지하면서도 무엇을 어느 단위에서 보호해야 하는가"를 실제 코드로 계속 검증해본 과정이라고 정리할 수 있습니다.  
아직 writer starvation이나 더 강한 조건 기반 락 같은 숙제는 남아 있지만, 반대로 말하면 이 시스템이 어디까지 왔고 다음에 어디를 봐야 하는지가 분명해진 상태라고 생각합니다.

## 발표 직전 체크리스트

- `idle timeout`은 아직 미구현이라고 정확히 말할 것
- `engine_lock`이 write DML 전체를 직렬화한다고 말하지 않을 것
- `thread-per-connection` 전환 이유를 "keep-alive와 worker 점유" 중심으로 설명할 것
- `row lock`과 `page latch`의 역할 차이를 분리해서 말할 것
- `scan 기반 UPDATE/DELETE` race는 "목록을 논리적으로 봉인했다"는 표현으로 설명할 것
- writer starvation은 숨기지 말고 다음 개선 포인트로 자연스럽게 연결할 것
