## 공동 질문 1. 클라이언트·서버 & 계층 모델

### 1. 클라이언트-서버 트랜잭션은 정확히 무엇인가? 요청, 처리, 응답, 완료 네 단계를 통틀어 트랜잭션이라고 부르는가? (최현진)

맞다. 클라이언트-서버 트랜잭션은 클라이언트가 서버에 요청을 보내고, 서버가 그 요청을 처리한 뒤, 응답을 돌려주고, 클라이언트가 그 응답을 받는 하나의 완결된 상호작용을 말한다. 웹에서는 브라우저가 HTTP 요청을 보내고 서버가 HTML, 이미지, JSON 같은 응답을 돌려주는 한 번의 왕복을 트랜잭션으로 볼 수 있다.

“완료”는 별도의 패킷 이름이라기보다, 클라이언트 입장에서 필요한 응답을 다 받았고 서버도 해당 요청 처리를 끝냈다는 논리적 상태다. TCP 연결은 트랜잭션 하나 뒤에 바로 닫힐 수도 있고, HTTP/1.1 keep-alive처럼 같은 연결에서 여러 트랜잭션이 이어질 수도 있다.

### 2. 네트워크는 계층적 구조를 가진다. TCP/IP 4계층을 기준으로 데이터를 보낼 때 어떤 과정을 거쳐서 네트워크에 데이터가 전송되는가? (이호준)

TCP/IP 4계층으로 보면 애플리케이션 데이터는 아래로 내려가며 계속 포장된다.

```text
Application: HTTP 요청, FTP 명령, DNS 질의 같은 실제 메시지
Transport: TCP/UDP 헤더 추가, 포트 번호와 신뢰성/순서 처리
Internet: IP 헤더 추가, 출발지/목적지 IP 주소와 라우팅 처리
Link: Ethernet/Wi-Fi 프레임 헤더 추가, MAC 주소 기준으로 다음 홉에 전달
```

예를 들어 브라우저가 `GET / HTTP/1.1`을 만들면 TCP는 이를 바이트 스트림으로 보고 TCP 세그먼트에 담는다. IP는 세그먼트를 IP 패킷에 담고 목적지 IP까지 가는 경로를 잡는다. 이더넷은 그 IP 패킷을 현재 링크에서 다음 장비로 보내기 위해 프레임에 담는다.

### 3. 클라이언트 컴퓨터에서 출발한 데이터가 네트워크에서 어떤 과정을 거쳐서 서버 컴퓨터에 도착하는가? (이호준)

클라이언트 애플리케이션이 `write()`나 `send()`를 호출하면 데이터는 사용자 공간에서 커널로 복사된다. 커널의 TCP/IP 스택은 TCP 헤더, IP 헤더, 링크 계층 헤더를 붙이고 NIC를 통해 프레임을 내보낸다.

프레임은 같은 LAN의 스위치나 공유기를 거쳐 라우터에 도착한다. 라우터는 링크 계층 헤더를 벗기고 IP 목적지를 확인한 뒤, 다음 네트워크로 보낼 새 링크 계층 헤더를 붙여 전달한다. 이런 홉을 반복해 서버가 있는 네트워크에 도착하면 서버 NIC가 프레임을 받고, 서버 커널이 TCP/IP 헤더를 해석해 최종적으로 서버 프로세스의 소켓 수신 버퍼에 데이터를 넣는다.

### 4. TCP/IP가 프로토콜 소프트웨어라면, 예전에 배운 encapsulation은 사라지는 것인가? (최현진)

사라지지 않는다. TCP/IP 프로토콜 소프트웨어가 바로 encapsulation을 구현한다. 애플리케이션 데이터에 TCP 헤더를 붙이고, 그 결과에 IP 헤더를 붙이고, 다시 링크 계층 헤더를 붙이는 동작이 캡슐화다.

프로토콜 소프트웨어라는 말은 이 작업이 커널의 네트워크 스택 코드로 구현되어 있다는 뜻이다. 개념적으로는 계층별 캡슐화이고, 구현적으로는 커널 자료구조와 함수들이 헤더를 만들고 검사하고 제거하는 과정이다.

### 5. IP는 인터넷 상위망에서 쓰이는 주소인데, 상위망에서 또 상위망으로 갈 때 따로 감싸는가? (최현진)

일반적인 라우팅에서는 IP 패킷 자체가 목적지까지 유지되고, 각 링크를 지날 때마다 링크 계층 프레임만 새로 만들어진다. 즉 라우터를 지날 때 IP 헤더는 TTL 같은 일부 필드만 바뀌고, Ethernet/Wi-Fi 같은 링크 계층 헤더는 매 홉 새로 붙는다.

```
IP Header의 TTL 1 감소 -> IP Payload 변경됨. -> IP Header Checksum 재계산.
```

다만 VPN, 터널링, MPLS 같은 기술에서는 “IP 패킷을 또 다른 IP 패킷이나 별도 헤더로 감싸는” 일이 있다. 하지만 CSAPP 11장의 기본 인터넷 흐름에서는 IP 위에 계속 IP를 중첩한다기보다, IP 패킷을 각 구간의 링크 프레임에 실어 나른다고 이해하면 된다.

### 6. Ethernet과 Internet은 무엇이 다른가? (최현진)

Ethernet은 주로 같은 LAN 안에서 프레임을 전달하는 링크 계층 기술이다. MAC 주소를 사용하고, 같은 네트워크 구간에서 NIC, 스위치, 공유기까지 데이터를 전달하는 데 쓰인다.

Internet은 여러 네트워크를 IP로 연결한 네트워크들의 네트워크다. IP 주소를 사용하고, 라우터들이 목적지 네트워크를 향해 패킷을 전달한다. 간단히 말하면 Ethernet은 한 링크 또는 LAN 내부 전달 방식이고, Internet은 수많은 LAN/WAN을 IP 라우팅으로 묶은 전체 구조다.

## 공동 질문 2. 네트워크 하드웨어 기초

### 1. 이더넷, 브릿지, 라우터는 무엇이 다른가. 이더넷과 브릿지는 대역폭만 다른가? (최우녕)

이더넷은 링크 계층의 통신 규격이고, 브릿지는 링크 계층에서 여러 이더넷 세그먼트를 연결하는 장비다. 현대 스위치는 다중 포트 브릿지라고 볼 수 있다. 브릿지는 MAC 주소를 보고 프레임을 어느 포트로 보낼지 결정한다.

라우터는 네트워크 계층 장비다. IP 주소와 라우팅 테이블을 보고 서로 다른 IP 네트워크 사이로 패킷을 전달한다. 따라서 이더넷과 브릿지는 대역폭만 다른 관계가 아니다. 이더넷은 규격, 브릿지는 그 규격의 프레임을 전달하는 장비, 라우터는 IP 패킷을 네트워크 사이에서 전달하는 장비다.

이더넷과 브릿지는 대역폭만 다르냐는 것은 잘못된 질문이다. 이더넷, 브릿지 차이가 아니라 해당 장비가 지원하는 대역폭과 랜선의 종류에 따라 다르다.

두 장비의 지원 대역폭이 다르다면, PHY(Physical Layer)계층에서 단순 전기 펄스 신호로 두 대역폭이 서로 지원 가능한 대역폭으로 맞춘다. 이것을 자동협상(auto-negotiation)이라고 한다.

### 2. 실제 생활에서 Ethernet, hub, bridge, router, LAN, WAN은 어떤 장비로 보면 되는가? (최현진)

Ethernet은 유선 랜 규격이라고 보면 된다. 집이나 회사의 랜 케이블과 랜 포트가 대표적이다. hub는 들어온 전기 신호를 모든 포트로 뿌리는 단순 장비이고, 요즘은 거의 스위치로 대체되었다. bridge는 MAC 주소를 보고 필요한 포트로만 프레임을 넘기는 장비이며, 현대의 Ethernet switch가 대표적이다.

router는 집의 공유기나 회사의 라우터처럼 서로 다른 네트워크를 연결하는 장비다. LAN은 집, 강의실, 회사 사무실 내부망 같은 가까운 범위의 네트워크이고, WAN은 ISP망이나 지사 간 연결처럼 넓은 범위의 네트워크다.

```
라우터 기능: 집 LAN <-> ISP 인터넷 연결
스위치 기능: LAN 포트 1, 2, 3, 4끼리 연결
Wi-Fi AP 기능: 무선 LAN 제공
NAT/방화벽 기능: 사설 IP <-> 공인 IP 변환

허브: 들어온 신호를 모든 포트로 뿌림 (그래서 의미 없어서 요즘 안 씀)
브릿지/스위치: MAC 주소 보고 같은 LAN 안에서 프레임 전달
라우터: IP 주소 보고 서로 다른 네트워크 사이에서 패킷 전달
```

### 3. 더미 허브가 무엇인가? (최현진)

**더미 허브 = 일반 허브**

더미 허브는 들어온 신호를 해석하지 않고 모든 포트로 그대로 복제해 내보내는 장비다. 목적지 MAC 주소를 학습하지 않기 때문에 같은 허브에 연결된 모든 호스트가 같은 충돌 영역을 공유한다.

그래서 한 장비가 보내는 동안 다른 장비가 동시에 보내면 충돌이 날 수 있고, 불필요한 트래픽도 모든 포트로 퍼진다. 스위치는 MAC 주소를 학습해 필요한 포트로만 프레임을 보내므로 더미 허브보다 효율적이다.

### 4. LAN과 WAN은 무엇이 다른가. (최우녕)

LAN은 제한된 지역 안의 네트워크다. 집, 연구실, 회사 사무실, 학교 강의실 내부망처럼 관리 범위가 작고 지연이 낮은 네트워크를 말한다. 보통 Ethernet이나 Wi-Fi가 LAN 기술로 쓰인다.

WAN은 넓은 지역을 연결하는 네트워크다. 통신사 백본망, 도시 간 회선, 국가 간 해저 케이블 같은 것이 WAN에 해당한다. WAN은 여러 LAN을 라우터와 통신사 인프라로 연결한다.

WAN은 공유기 바깥쪽, ISP/인터넷 쪽. 사용기술: Ethernet, 광전송망, MPLS, PPP, DOCSIS, LTE/5G, 위성 링크 ...

### 5. LAN과 WAN은 상대적인 말인가, 절대적인 말인가? (최현진)

어느 정도 상대적인 말이다. 핵심은 물리적 거리만이 아니라 관리 범위, 기술, 지연, 소유 주체다. 한 회사 캠퍼스 전체가 큰 LAN처럼 운영될 수도 있고, 지사 간 연결은 같은 회사 네트워크라도 WAN으로 볼 수 있다.

그래도 일반적인 기준은 있다. 같은 건물이나 가까운 지역의 내부망은 LAN, 통신사망이나 먼 지역을 연결하는 네트워크는 WAN이라고 부른다.

### 6. Ethernet adapter가 주소를 가지는 이유는 무엇인가? (최현진)

Ethernet adapter = MAC address / 다음 홉을 찾기 위함. 따라서 홉을 옮길 때마다 이 Ethernet 캡슐이 변경된다.

같은 링크 안에서 프레임을 누구에게 보낼지 식별하기 위해서다. Ethernet adapter, 즉 NIC는 MAC 주소를 가진다. Ethernet 프레임에는 출발지 MAC 주소와 목적지 MAC 주소가 들어가고, 스위치와 NIC는 이를 보고 프레임을 전달하거나 수신한다.

IP 주소가 최종 목적지 네트워크 계층 주소라면, MAC 주소는 현재 LAN 구간에서 다음 장비까지 전달하기 위한 링크 계층 주소다.

### 7. frame과 packet은 하는 일이 비슷한데, 둘은 무엇이 다른가? (최현진)

packet은 보통 네트워크 계층의 IP 패킷을 말하고, frame은 링크 계층의 Ethernet/Wi-Fi 프레임을 말한다. IP 패킷에는 출발지/목적지 IP 주소가 들어가며 여러 네트워크를 지나 최종 목적지까지 가는 논리적 단위다.

```
IP Header의 TTL 1 감소 -> IP Payload 변경됨. -> IP Header Checksum 재계산.
```

프레임은 한 링크 구간에서 다음 장비까지 전달되는 단위다. 프레임 안에 IP 패킷이 payload로 들어갈 수 있다. 라우터를 지날 때 IP 패킷은 유지되지만 프레임 헤더는 다음 링크에 맞게 매번 바뀐다.

### 8. 호스트들은 결국 비트를 주고받는다고 하는데, 그 “비트 뭉치”는 어떤 형태이며 어떻게 수신자가 자기 데이터라는 것을 아는가. (최우녕)

물리적으로는 전기 신호, 빛 신호, 전파 신호의 변화로 비트가 전달된다. 논리적으로는 이 비트들이 Ethernet frame, IP packet, TCP segment 같은 구조화된 단위로 해석된다.

수신자는 프레임의 목적지 MAC 주소를 보고 현재 링크에서 자신에게 온 프레임인지 판단한다. 그 안의 IP 헤더 목적지 IP가 자기 주소인지 확인하고, TCP/UDP 헤더의 포트 번호를 보고 어느 소켓 또는 프로세스로 전달할지 결정한다. 즉 “나의 데이터인가”는 MAC, IP, port를 단계적으로 검사하며 결정된다.

+++
인터넷 선은 클럭(clock)을 사용하지 않는다.
- 선이 두 배로 늘어난다.
- 고속/장거리 통신에서 클럭 신호와 데이터 신호의 싱크가 안 맞을 수 있다. (aka. skew 문제)

클럭을 사용하지 않고 패턴을 이용하여 해석하는 기술을 사용한다. 이를 clock recovery기술이라한다. 클럭보다 당연히 비효율적이지만 위의 이유 때문에 안정성이 높다.

LAN은 단순한 1과 0으로 이루어지지 않는다. 전압의 차이를 이용한다. 따라서 이론적으론 데이터 통신 전선 하나, 전압 기준용 GND선 하나만 있으면 통신이 가능함.(송수신 둘 다 되려면 통신 전선 두 개)

하지만, Ethernet은 두 선의 전압의 차이를 이용한다. 이 방식은 외부 노이즈에 강해서 Ethernet은 꼬임쌍선 즉, twisted pair를 사용한다. 꼬임쌍선을 사용하는 이유는 전자기 유도로 인한 두 전류가 영향을 주지 않게 하기 위함. 1쌍은 Tx송신, 1쌍은 Rx수신.

그리고 사실 각 쌍선은 송 수신을 동시에 한다. 자기 자신이 보낸 신호를 알고있으면 그 신호를 뺌으로써 상대 신호를 해석할 수 있기 때문. (echo cancellation)

## 공동 질문 3. IP 주소 기본

### 1. IPv4와 IPv6의 주소 형식과 크기는 어떻게 다른가. (최우녕)

/

IPv4는 32비트 주소이고 보통 `192.168.0.1`처럼 8비트씩 4개로 나누어 10진수로 표기한다. 가능한 주소 수는 약 43억 개다.

IPv6는 128비트 주소이고 보통 `2001:db8::1`처럼 16비트 단위 8개를 16진수로 표기한다. 주소 공간이 매우 커서 NAT 없이도 훨씬 많은 장비에 고유 주소를 줄 수 있고, 자동 설정과 확장 헤더 같은 기능도 고려되어 있다.

### 2. IPv4는 32비트 정수 한 개라는데, 어떻게 그것만으로 전 세계 주소를 표현하는가. (최우녕)

/

IPv4 주소는 32비트 숫자 하나지만, 이 숫자는 계층적으로 배정된다. 각 국가, ISP, 회사, 클라우드 사업자 등은 주소 블록을 할당받고, 그 내부에서 다시 네트워크와 호스트 부분을 나누어 사용한다.

다만 32비트 주소 공간은 실제로 부족하다. 그래서 사설 IP, NAT, CIDR 같은 기술이 널리 쓰이고, 더 큰 주소 공간을 제공하는 IPv6가 등장했다. “32비트로 전 세계를 표현한다”는 것은 모든 장비가 동시에 고유 공인 IPv4를 넉넉히 가진다는 뜻이 아니라, 제한된 공간을 계층 할당과 NAT로 운영해 왔다는 뜻이다.

### 3. 111.111... 같은 IP 주소에 프로토콜 소프트웨어의 핵심 기능인 주소 정보가 들어 있으니, 프로그래머가 복잡한 코드를 짤 필요가 없다는 뜻인가? (최현진)

/

일부는 맞지만 전부는 아니다. IP 주소가 목적지 정보를 담고 있고, 커널 TCP/IP 스택과 라우터가 이를 보고 전달해 주기 때문에 애플리케이션 프로그래머가 직접 라우팅 알고리즘을 구현할 필요는 없다.

하지만 프로그래머는 여전히 주소 해석, 포트 선택, 소켓 생성, 연결, 에러 처리, 부분 읽기/쓰기, 타임아웃, 프로토콜 메시지 형식 같은 코드를 작성해야 한다. IP 주소는 네트워크 계층의 전달에 필요한 핵심 정보이지, 애플리케이션 통신 전체를 자동으로 해결하는 정보는 아니다.

## 공동 질문 4. DNS 기본

### 1. DNS는 무엇이고 어떤 용도로 쓰이는가. (최우녕)

/

DNS는 Domain Name System의 약자로, 사람이 읽기 쉬운 도메인 이름을 IP 주소 같은 네트워크 주소 정보로 바꿔 주는 분산 데이터베이스다. 예를 들어 브라우저가 `example.com`에 접속하려면 먼저 DNS를 통해 해당 이름의 IP 주소를 알아낸다.

DNS는 단순히 이름을 IP로 바꾸는 것뿐 아니라 메일 서버 정보(MX), 별칭(CNAME), 텍스트 인증 정보(TXT), 여러 IP로의 부하 분산 등에도 쓰인다. 네트워크 입장에서는 IP 주소가 필요하지만, 사람과 서비스 운영 입장에서는 도메인 이름이 훨씬 관리하기 쉽기 때문에 DNS가 필요하다.

## 공동 질문 5. Sockets Interface 호출 순서

### 1. 11.4장에 등장하는 함수와 호출 순서를 정리해 달라. / q07의 socket interface 호출 순서는 이미 앞에서 다 배운 내용인가? (최우녕, 최현진)

/

클라이언트의 기본 순서는 `getaddrinfo -> socket -> connect -> read/write -> close`다. `getaddrinfo`로 서버 주소 후보를 얻고, `socket`으로 통신 끝점을 만들고, `connect`로 서버에 TCP 연결을 맺은 뒤 데이터를 주고받는다.

서버의 기본 순서는 `getaddrinfo -> socket -> setsockopt -> bind -> listen -> accept -> read/write -> close`다. `bind`로 로컬 주소와 포트에 소켓을 묶고, `listen`으로 연결 요청을 받을 준비를 하며, `accept`로 실제 클라이언트별 connected socket을 얻는다. 이 흐름은 유닉스 I/O의 fd 개념 위에 네트워크 주소와 연결 절차가 추가된 내용이다.

### 2. getaddrinfo를 먼저 부른 뒤 socket, connect를 차례로 실행한다는 말은 구체적으로 어떤 코드인가. (최우녕)

/

대표 코드는 다음과 같다.

```c
struct addrinfo hints, *listp, *p;
int clientfd;

memset(&hints, 0, sizeof(hints));
hints.ai_socktype = SOCK_STREAM;
hints.ai_flags = AI_NUMERICSERV;
hints.ai_flags |= AI_ADDRCONFIG;

getaddrinfo(hostname, port, &hints, &listp);

for (p = listp; p; p = p->ai_next) {
    clientfd = socket(p->ai_family, p->ai_socktype, p->ai_protocol);
    if (clientfd < 0)
        continue;
    if (connect(clientfd, p->ai_addr, p->ai_addrlen) != -1)
        break;
    close(clientfd);
}

freeaddrinfo(listp);
```

`getaddrinfo`는 가능한 주소 후보 목록을 만들고, 각 후보에 대해 소켓을 만든 뒤 `connect`를 시도한다. 하나가 성공하면 그 fd로 서버와 통신한다.

### 3. 서버에서 socket -> bind -> listen -> accept가 각각 어떤 역할인가? (최현진)

/

`socket()`은 커널에 소켓 객체를 만들고 fd를 반환한다. 아직 주소나 포트에 묶인 상태는 아니다. `bind()`는 이 소켓을 특정 로컬 IP와 포트에 연결한다. 예를 들어 서버가 8080 포트에서 요청을 받게 만든다.

`listen()`은 이 소켓을 수동 열기 상태로 바꿔 TCP 연결 요청을 받을 수 있게 한다. 이때의 소켓은 listening socket이다. `accept()`는 대기 중인 클라이언트 연결 하나를 꺼내고, 그 클라이언트와 실제로 통신할 connected socket fd를 새로 반환한다.

## 공동 질문 6. TCP vs UDP 기초

### 1. TCP와 UDP의 소켓 함수는 “시스템 콜로 구현되어 있다”는데, 이게 어떤 구조인가. (최우녕)

/

사용자 프로그램은 `socket`, `connect`, `bind`, `listen`, `accept`, `send`, `recv` 같은 함수를 호출한다. 이 함수들은 libc 래퍼를 거쳐 커널 시스템 콜로 진입한다. 커널 안에서는 fd table을 통해 소켓 객체를 찾고, 그 소켓의 프로토콜이 TCP인지 UDP인지에 따라 다른 함수들이 실행된다.

즉 API 모양은 비슷하지만 내부 구현은 프로토콜별로 분기된다. TCP 소켓이면 연결 상태, 순서 번호, 재전송, 흐름 제어를 처리하고, UDP 소켓이면 연결 없이 데이터그램 송수신을 처리한다.

### 2. “TCP는 host-to-host, UDP는 process-to-process”라는 표현은 어떤 의미인가. 보통은 반대로 들리는데 왜 그렇게 말하는가. (최우녕)

/

엄밀히 말하면 IP가 host-to-host 전달을 담당하고, TCP와 UDP는 포트 번호를 사용해 process-to-process 전달을 제공한다. 따라서 “TCP는 host-to-host, UDP는 process-to-process”라고 단정하면 혼동될 수 있다.

다만 문맥에 따라 TCP를 “호스트 사이에 신뢰성 있는 연결을 만든다”고 표현할 수는 있다. 그러나 실제 TCP 연결도 4-tuple, 즉 출발지 IP/포트와 목적지 IP/포트로 식별되므로 프로세스의 소켓까지 구분한다. UDP도 포트로 프로세스까지 전달하지만 연결 상태와 신뢰성은 제공하지 않는다.

### 3. TCP와 UDP의 공통점과 차이를 한 번에 정리해 달라. (최우녕)

/

공통점은 둘 다 IP 위에서 동작하는 전송 계층 프로토콜이고, 포트 번호를 사용해 한 호스트 안의 어떤 프로세스 또는 소켓으로 데이터를 보낼지 구분한다는 점이다.

차이는 크다. TCP는 연결 지향, 바이트 스트림, 신뢰성, 순서 보장, 재전송, 흐름 제어, 혼잡 제어를 제공한다. UDP는 비연결, 데이터그램 단위, 순서/도착/중복 제거를 보장하지 않는 단순한 전송을 제공한다. TCP는 안정적인 웹, 파일 전송, SSH에 적합하고, UDP는 DNS, 실시간 스트리밍, 게임처럼 지연과 단순성이 중요한 곳에 적합하다.

### 4. TCP와 UDP의 전송 방식 차이를 비유하면 무엇인가? (최현진)

/

TCP는 전화 통화나 택배 추적 서비스에 가깝다. 먼저 연결을 맺고, 보낸 순서를 맞추고, 빠진 데이터가 있으면 다시 보내며, 상대가 받을 수 있는 속도도 고려한다.

UDP는 엽서나 확성기 방송에 가깝다. 하나의 메시지를 바로 보내지만, 도착 여부와 순서를 기본적으로 보장하지 않는다. 대신 구조가 단순하고 지연이 작다.

## 공동 질문 7. 4관점 통합

### 1. 네트워크 통신 한 번을 CPU, 메모리, 커널, 파일 핸들(= fd)의 관점으로 각각 나눠서 설명해 달라. / network 통신 한 번을 CPU, memory, kernel, fd 관점으로 나누어 본다는 것은 무슨 의미인가? (최우녕, 최현진)

/

fd 관점에서는 소켓도 파일처럼 정수 핸들로 다루어진다. 애플리케이션은 `read(sockfd, ...)`, `write(sockfd, ...)`처럼 호출하고, 커널은 fd table을 통해 실제 소켓 객체를 찾는다.

커널 관점에서는 시스템 콜 진입 후 TCP/IP 스택, 라우팅, 소켓 버퍼, NIC 드라이버가 동작한다. 메모리 관점에서는 사용자 버퍼와 커널 버퍼 사이 복사, page cache나 socket buffer, DMA 가능한 메모리 등이 중요하다. CPU 관점에서는 사용자 코드 실행, 커널 코드 실행, 인터럽트 처리, 체크섬 계산, 프로토콜 처리, 컨텍스트 스위칭 비용이 발생한다.

### 2. 송신과 수신을 대칭으로 묘사해 달라. (최우녕)

/

송신은 사용자 버퍼에서 시작해 `write()`로 커널에 들어가고, 커널 소켓 버퍼와 TCP/IP 스택을 지나 NIC 드라이버와 NIC를 통해 네트워크로 나간다.

수신은 반대다. NIC가 프레임을 받고 DMA로 메모리에 올린 뒤 인터럽트나 polling으로 커널에 알린다. 커널은 링크/IP/TCP 헤더를 해석하고 데이터를 해당 소켓 수신 버퍼에 넣는다. 사용자 프로세스가 `read()`를 호출하면 커널 수신 버퍼의 데이터가 사용자 버퍼로 복사된다.

### 3. 이 관점에서 성능에 영향을 주는 요소는 무엇인가. (최우녕)

/

CPU 관점에서는 시스템 콜 횟수, 컨텍스트 스위칭, 인터럽트 처리, 체크섬과 암호화 비용이 중요하다. 메모리 관점에서는 사용자-커널 복사 횟수, 캐시 locality, 버퍼 크기, zero-copy 가능 여부가 영향을 준다.

커널 관점에서는 TCP 혼잡 제어, socket buffer 크기, 큐잉 지연, epoll 같은 이벤트 처리 방식, NIC 드라이버 성능이 중요하다. fd 관점에서는 너무 많은 fd를 다룰 때 select의 한계, epoll 사용 여부, fd 누수 여부가 성능과 안정성에 영향을 준다.

## 공동 질문 8. HTTP 개론

### 1. FTP는 무엇이고 HTTP와 무엇이 다른가. / HTTP와 FTP는 무엇이 다르고, 왜 HTTP가 웹에 더 잘 맞는가? (최우녕, 최현진)

/

FTP는 파일 전송을 위한 프로토콜이다. 전통적인 FTP는 명령을 주고받는 control connection과 실제 데이터를 보내는 data connection을 따로 사용한다. 로그인, 디렉터리 이동, 파일 업로드/다운로드 같은 기능이 중심이다.

HTTP는 문서와 리소스를 요청하고 응답받는 웹 프로토콜이다. 요청 하나와 응답 하나의 구조가 단순하고, HTML 문서 안에서 이미지, CSS, JS 같은 여러 리소스를 링크할 수 있다. 브라우저, 캐시, 프록시, CDN과 결합하기 쉬워 웹에는 HTTP가 더 잘 맞는다.

### 2. FTP는 무엇이고, HTTP/HTML의 강력함을 설명하려고 나온 비교 대상인가? (최현진)

/

CSAPP에서 FTP는 HTTP와 비교해 인터넷 프로토콜의 성격을 설명하기 좋은 예로 등장한다. FTP도 TCP 위에서 동작하는 응용 프로토콜이고 파일 전송에는 적합하지만, 웹처럼 문서들이 링크로 연결되고 브라우저가 여러 리소스를 자동으로 가져오는 모델과는 다르다.

HTTP/HTML의 강력함은 HTTP가 리소스 요청/응답을 단순하게 만들고, HTML이 다른 리소스를 참조할 수 있게 해 웹 페이지라는 복합 문서를 구성한다는 데 있다. FTP는 이와 대비되는 파일 중심 프로토콜로 보면 된다.

### 3. 정적 콘텐츠와 동적 콘텐츠는 실제로 무엇이 다른가? (최현진)

/

정적 콘텐츠는 서버 디스크에 이미 존재하는 파일을 거의 그대로 보내는 것이다. 예를 들어 `.html`, `.css`, `.jpg` 파일을 읽어 HTTP 응답 본문으로 보내는 경우다.

동적 콘텐츠는 요청 시점에 프로그램을 실행하거나 DB를 조회해 새로 만든 응답이다. 예를 들어 `/cgi-bin/adder?1&2` 요청을 받으면 서버가 CGI 프로그램을 실행해 계산 결과 HTML을 생성하는 식이다. 차이는 “파일을 그대로 주느냐”와 “요청에 따라 실행 결과를 만들어 주느냐”다.

## 공동 질문 9. Tiny Web Server

### 1. 책에서 말하는 Tiny 서버는 무엇인가. 최소한 무엇을 할 수 있어야 Tiny인가. (최우녕)

/

Tiny는 CSAPP에서 교육용으로 제시하는 작은 웹 서버다. HTTP 요청을 받고, 요청 URI를 해석하며, 정적 파일을 반환하거나 CGI 프로그램을 실행해 동적 콘텐츠를 반환할 수 있다.

최소한 listening socket을 열고, 클라이언트 연결을 accept하고, HTTP request line과 header를 읽고, URI가 정적인지 동적인지 구분한 뒤, 올바른 HTTP 응답 헤더와 본문을 보내야 Tiny라고 할 수 있다.

### 2. Tiny를 구성하는 함수(main, doit, read_requesthdrs, parse_uri, serve_static, get_filetype, serve_dynamic, clienterror)가 각각 어떤 역할을 하고 어떻게 호출되는가. (최우녕)

/

`main`은 포트를 받아 listening socket을 만들고, 반복적으로 `accept`한 뒤 각 연결을 `doit`에 넘긴다. `doit`은 연결 하나의 HTTP 트랜잭션을 처리하는 핵심 함수로, request line을 읽고 method와 URI를 파싱한다.

`read_requesthdrs`는 요청 헤더를 읽어 버린다. `parse_uri`는 URI가 정적 콘텐츠인지 동적 콘텐츠인지 판단하고 파일 이름과 CGI 인자를 만든다. 정적이면 `serve_static`이 파일 타입을 `get_filetype`으로 판단한 뒤 파일 내용을 응답한다. 동적이면 `serve_dynamic`이 fork/exec로 CGI 프로그램을 실행한다. 오류가 있으면 `clienterror`가 HTTP 오류 응답을 만든다.

### 3. Tiny의 정적 / 동적 처리 흐름을 코드와 함께 따라가 달라. (최우녕)

/

정적 처리 흐름은 `accept -> doit -> parse_uri -> stat -> serve_static`이다. `serve_static`은 파일을 열고, `Content-length`와 `Content-type` 헤더를 보낸 뒤 파일 내용을 클라이언트 fd로 복사한다.

동적 처리 흐름은 `accept -> doit -> parse_uri -> stat -> serve_dynamic`이다. `serve_dynamic`은 먼저 HTTP 응답 시작 줄을 보내고, 자식 프로세스를 `fork()`한다. 자식은 `QUERY_STRING` 환경변수를 설정하고 `dup2(fd, STDOUT_FILENO)`로 표준 출력을 클라이언트 소켓에 연결한 뒤 `execve()`로 CGI 프로그램을 실행한다. CGI 프로그램이 `printf()`한 내용은 표준 출력으로 나가고, 실제로는 클라이언트 소켓으로 전달된다.

# D Questions

## D-1. Unix I/O & FD 추상화

### 1. VFS 네 개의 핵심 객체(superblock / inode / dentry / file)는 어떤 관계이고 커널에서 어떻게 표현되나? (최우녕)

VFS는 서로 다른 파일 시스템을 같은 인터페이스로 다루기 위한 커널의 추상화 계층이다.

- `superblock`: 마운트된 파일 시스템 전체의 메타데이터를 나타낸다. 예를 들어 ext4 파티션 하나가 마운트되면 그 파일 시스템의 블록 크기, 루트 inode, 동작 함수 테이블 등을 가진다.
- `inode`: 파일 하나의 실체를 나타낸다. 파일 타입, 권한, 소유자, 크기, 디스크 블록 위치 같은 메타데이터를 가진다. 파일 이름은 inode에 직접 저장되지 않는다.
- `dentry`: 디렉터리 엔트리, 즉 “이 이름이 어떤 inode를 가리키는가”를 캐시하는 객체다. `/home/a.txt`에서 `home`, `a.txt` 같은 경로 이름 해석에 쓰인다.
- `file`: 프로세스가 `open()`으로 실제로 연 “열린 파일 상태”다. 현재 파일 오프셋, open flag, 접근 모드, file operation table 포인터 등을 가진다.

관계는 대략 `fd -> file -> dentry -> inode -> superblock`이다. 프로세스는 정수 fd만 알고, 커널은 fd table을 통해 `struct file`을 찾고, 그 뒤 dentry와 inode를 따라 실제 파일 객체에 도달한다.

File system
```
ext4   : 리눅스 디스크 파일 시스템
xfs    : 대용량 서버에서 많이 쓰는 파일 시스템
btrfs  : 스냅샷/체크섬 기능이 강한 파일 시스템
tmpfs  : 메모리 기반 파일 시스템
procfs : 커널 정보를 파일처럼 보여 주는 가상 파일 시스템
sysfs  : 장치/커널 객체 정보를 파일처럼 보여 주는 가상 파일 시스템
nfs    : 네트워크 파일 시스템
```

### 2. 디스크 위의 ext4는 실제로 어떤 레이아웃으로 쓰여 있나? (최우녕)

ext4는 디스크를 여러 블록 그룹으로 나누어 관리한다. 전체 구조는 보통 다음과 같다.

```text
슈퍼블록
블록 그룹 디스크립터
블록 비트맵
inode 비트맵
inode 테이블
데이터 블록들
```

슈퍼블록은 파일 시스템 전체 정보를 담고, 블록 그룹 디스크립터는 각 그룹의 위치와 상태를 담는다. 비트맵은 어떤 블록과 inode가 사용 중인지 표시한다. inode 테이블에는 파일 메타데이터가 저장되고, 데이터 블록에는 파일 내용이나 디렉터리 엔트리가 저장된다. ext4는 저널링도 사용하므로, 장애가 나도 메타데이터 갱신을 비교적 안전하게 복구할 수 있다.

### 3. `open("/home/woonyong/a.txt")`는 커널 안에서 몇 단계를 거치나? (최우녕)

대표 흐름은 다음과 같다.

1. 사용자 프로세스가 `open()` 시스템 콜을 호출한다.
2. 커널이 사용자 문자열 경로를 안전하게 복사한다.
3. VFS가 `/`부터 시작해 `home`, `woonyong`, `a.txt`를 차례로 dentry cache에서 찾는다.
4. 캐시에 없으면 해당 디렉터리의 inode와 파일 시스템 구현을 통해 디스크에서 디렉터리 엔트리를 읽는다.
5. 최종 inode를 찾고 권한을 검사한다.
6. `struct file`을 만든다.
7. 현재 프로세스의 fd table에서 빈 번호를 찾고 그 번호가 `struct file`을 가리키게 한다.
8. 사용자에게 fd 정수를 반환한다.

즉 `open()`의 결과는 파일 내용을 읽어 오는 것이 아니라, “이 프로세스가 이 열린 파일 객체를 fd 번호로 사용할 수 있다”는 연결을 만드는 것이다.

### 4. `read()` 한 번이 VFS → FS → page cache → block layer → 디스크로 어떻게 흐르나? (최우녕)

`read(fd, buf, n)`을 호출하면 커널은 fd table에서 `struct file`을 찾고, 그 파일의 연산 함수로 읽기를 수행한다. 일반 파일이면 VFS가 파일 시스템 구현(ext4 등)의 read 경로로 내려간다.

먼저 page cache를 확인한다. 요청한 파일 데이터가 메모리 캐시에 있으면 디스크를 읽지 않고 곧바로 사용자 버퍼로 복사한다. 없으면 page fault 또는 read 경로에서 디스크 I/O가 발생한다. 이때 파일 시스템은 inode의 블록 매핑 정보를 보고 “파일의 이 오프셋은 디스크의 어떤 블록인가”를 계산하고, block layer에 요청을 넘긴다. block layer는 요청을 병합·정렬해 디바이스 드라이버로 보내고, 디스크나 SSD가 데이터를 읽어 오면 page cache에 채운 뒤 사용자 버퍼로 복사한다.

핵심은 `read()`가 매번 디스크를 직접 읽는 함수가 아니라는 점이다. 대부분은 page cache를 먼저 거치며, 실제 디스크 I/O는 캐시에 없을 때만 발생한다.

### 5. 소켓·파이프·procfs가 왜 “파일”처럼 보이나? (최우녕)

유닉스의 “모든 것은 파일”은 모든 대상이 디스크 파일이라는 뜻이 아니다. 다양한 I/O 대상을 fd라는 공통 핸들로 다루고, `read`, `write`, `close`, `poll` 같은 공통 시스템 콜을 적용할 수 있다는 뜻이다.

소켓은 네트워크 연결의 끝점이고, 파이프는 커널 안의 바이트 스트림 버퍼이며, procfs는 커널 상태를 파일처럼 보여 주는 가상 파일 시스템이다. 내부 구현은 모두 다르지만, 커널은 각 객체마다 다른 file operation table을 연결한다. 그래서 사용자 입장에서는 fd에 대해 `read(fd, ...)`를 호출하면 되고, 실제 동작은 커널 내부의 객체 타입별 함수가 처리한다.

### 6. 모든 외부 장치를 하나의 I/O로 보고 똑같이 처리하는 시스템을 만든 것인가? (최현진)

거의 맞다. 유닉스는 디스크 파일, 터미널, 파이프, 소켓, 디바이스를 모두 fd 기반 I/O 모델로 통합했다. 하지만 “완전히 똑같다”는 뜻은 아니다. 공통 진입점이 같을 뿐, 객체마다 가능한 연산과 의미는 다르다.

예를 들어 일반 파일은 seek가 가능하지만 TCP 소켓은 seek가 불가능하다. 디스크 파일의 `read()`는 파일 오프셋에서 데이터를 읽지만, 소켓의 `read()`는 수신 버퍼에서 도착한 바이트를 꺼낸다. 따라서 같은 fd 인터페이스를 쓰되, 객체별 의미는 file operation table에 의해 달라진다고 이해하면 된다.

### 7. 소켓도 결국 파일이라는 말 때문에, 모든 외부 장치를 내용이 계속 업데이트되는 USB처럼 생각해도 되는가? (최현진)

그렇게 생각하면 오해가 생긴다. 소켓은 저장 장치처럼 “어딘가에 고정된 내용이 있고 그걸 읽는다”는 모델이 아니다. 소켓은 커널 안의 네트워크 송수신 상태, 수신 큐, 송신 큐, TCP 상태 등을 가진 통신 끝점이다.

USB 저장 장치의 파일은 디스크 블록에 내용이 저장되어 있고 나중에 다시 읽을 수 있다. 반면 TCP 소켓에서 한 번 읽은 데이터는 수신 큐에서 소비된다. 파이프도 마찬가지로 저장 파일이 아니라 흐르는 바이트 스트림이다. “파일처럼 보인다”는 말은 fd와 `read/write/close` 인터페이스를 공유한다는 뜻이지, 저장 장치라는 뜻은 아니다.

### 8. 소켓은 주소가 저장되는 구조체인가, 파일 같은 것인가? (최현진)

둘 다 일부 맞지만 정확히는 “fd로 접근하는 커널 객체이며, 그 내부에 주소와 프로토콜 상태가 들어 있다”가 맞다.

사용자 프로세스는 `socket()`의 반환값인 fd를 받는다. 이 fd는 fd table을 통해 `struct file`을 가리키고, 그 뒤에 커널의 socket 객체와 TCP/UDP별 상태 객체가 연결된다. 그 안에 로컬 IP, 로컬 포트, 원격 IP, 원격 포트, 연결 상태, 송수신 버퍼 등이 저장된다. 따라서 사용자에게는 파일 같은 fd로 보이지만, 내부 실체는 네트워크 통신 상태를 가진 소켓 자료구조다.

### 9. fd는 무엇이고 책 어디에서 나오는가? (최현진)

fd(file descriptor)는 프로세스가 열린 I/O 객체를 가리키기 위해 사용하는 작은 정수다. CSAPP에서는 유닉스 I/O를 설명하는 10장과 네트워크 프로그래밍을 설명하는 11장에서 계속 등장한다. 파일을 열면 fd가 나오고, 소켓을 만들어도 fd가 나온다.

중요한 점은 fd 자체가 파일 객체가 아니라는 것이다. fd는 현재 프로세스의 fd table 인덱스다. 같은 숫자 3이라도 프로세스 A의 3번과 프로세스 B의 3번은 서로 다른 객체를 가리킬 수 있다.

### 10. FDT는 뭐라고 부르면 되고, 정확히 무엇인가? (최현진)

FDT는 file descriptor table, 즉 파일 디스크립터 테이블이라고 부르면 된다. 프로세스마다 존재하는 테이블이며, fd 정수를 `struct file *` 같은 커널 객체 참조로 바꿔 주는 배열이라고 볼 수 있다.

예를 들어 어떤 프로세스에서 `fd = 3`이면 커널은 그 프로세스의 fd table의 3번 칸을 본다. 그 칸이 어떤 `struct file`을 가리키면 read/write/close 같은 작업이 그 객체에 대해 수행된다. fd table은 프로세스별로 분리되어 있기 때문에 각 프로세스는 독립적인 fd 번호 공간을 가진다.

### 11. FD table에서 file table entry 주소를 어떻게 찾아가는거지? + file table, v-node에 대한 정보는 어디서 가지고 오는거지? / fd, v-node, i-node의 연결 관계는 어떻게 되는가? (정범진, 최현진)

CSAPP식 모델로 보면 다음과 같다.

```text
프로세스 fd table
  fd 3
   |
   v
open file table entry
  현재 파일 위치, 참조 횟수, 접근 모드
   |
   v
v-node / inode
  파일 타입, 권한, 크기, 디스크 위치
```

리눅스식으로 더 구체화하면 `fd -> struct file -> dentry -> inode` 흐름이다. fd table은 프로세스 안에 있고, `struct file`은 “열린 파일 상태”를 나타내며, inode는 파일 자체의 메타데이터를 나타낸다. 같은 파일을 두 번 `open()`하면 inode는 같아도 `struct file`은 보통 두 개가 생긴다. 반면 `fork()`로 fd가 복제되면 부모와 자식이 같은 `struct file`을 공유할 수 있다.

### 12. 식별자 테이블을 프로세스마다 가지는 이유는 무엇이고, 결국 fd와 FDT는 무엇을 하려고 있는가? (최현진)

프로세스마다 독립적인 실행 환경을 제공하기 위해서다. 각 프로세스는 자기 기준으로 0, 1, 2, 3 같은 작은 fd 번호를 사용한다. 커널 내부 포인터나 전역 객체 주소를 사용자에게 직접 노출하지 않아도 되고, 권한과 수명을 프로세스 단위로 관리할 수 있다.

fd와 FDT의 목적은 “사용자 프로그램이 작은 정수 핸들로 커널 I/O 객체를 안전하게 참조하게 하는 것”이다. 덕분에 C 프로그램은 파일이든 소켓이든 `read(fd, ...)`처럼 단순한 인터페이스로 다룰 수 있고, 커널은 fd table을 통해 실제 객체를 찾아 적절한 함수를 호출한다.

### 13. stdin/stdout/stderr가 “꼭 0/1/2여야” 하는 이유가 있나? 커널이 fd 번호를 구분하나? (최우녕)

커널이 0, 1, 2를 특별한 의미로 강제하는 것은 아니다. 이것은 유닉스 계열에서 정한 관례다.

- fd 0: 표준 입력
- fd 1: 표준 출력
- fd 2: 표준 에러

셸과 libc와 많은 프로그램이 이 관례를 따른다. 프로그램이 시작될 때 셸은 보통 0, 1, 2를 터미널이나 파일, 파이프에 연결해 둔다. 커널 입장에서는 그냥 fd table의 0번, 1번, 2번 슬롯일 뿐이다. 그래서 `dup2()`로 1번을 파일에 연결하면 `printf()` 출력이 파일로 가는 리다이렉션이 가능하다.

### 14. 3번 fd에 열린 소켓을 0/1번에 꽂으려면 어떻게 해야 하나? (최우녕)

`dup2()`를 사용한다.

```c
dup2(sockfd, 0); // 표준 입력을 소켓으로 변경
dup2(sockfd, 1); // 표준 출력을 소켓으로 변경
```

이렇게 하면 이후 프로그램이 `read(0, ...)`을 호출할 때 실제로는 소켓에서 읽고, `write(1, ...)` 또는 `printf()`를 호출할 때 실제로는 소켓으로 쓴다. CGI, 셸 리다이렉션, inetd 스타일 서버가 이런 방식으로 “프로그램은 표준 입출력만 쓰는데 실제 연결은 소켓”인 구조를 만든다.

### 15. “unbuffered”, “line-buffered”, “flush”가 무슨 소리인가? 커널 버퍼? libc 버퍼? (최우녕)

여기서 말하는 버퍼링은 보통 libc의 stdio 버퍼링을 의미한다. `printf`, `fgets`, `fprintf` 같은 함수는 바로 `write()` 시스템 콜을 호출하지 않고, 사용자 공간의 `FILE` 버퍼에 데이터를 모아 둘 수 있다.

- unbuffered: 거의 모아 두지 않고 즉시 출력한다. 보통 `stderr`가 여기에 가깝다.
- line-buffered: 줄바꿈 문자가 나올 때 버퍼를 비운다. 터미널에 연결된 `stdout`이 보통 그렇다.
- block-buffered: 버퍼가 일정 크기 찰 때까지 모았다가 출력한다. 파일로 리다이렉션된 `stdout`이 보통 그렇다.
- flush: libc 버퍼에 쌓인 데이터를 커널로 밀어 넣는 동작이다. `fflush()`가 여기에 해당한다.

주의할 점은 `fflush()`가 디스크에 영구 저장까지 보장하는 것은 아니라는 점이다. `fflush()`는 libc 버퍼에서 커널로 보내는 것이고, 디스크 반영까지 강하게 요구하려면 `fsync()` 같은 호출이 필요하다.

### 16. rio 함수들은 일반적인 file 함수와 어떤 차이점을 가지고 있는거지? (정범진)

CSAPP의 RIO(Robust I/O)는 짧은 읽기와 인터럽트 같은 상황을 안전하게 처리하기 위한 래퍼다. 일반 `read()`는 요청한 바이트 수보다 적게 읽고 반환할 수 있다. 특히 소켓, 파이프, 터미널에서는 이것이 정상적인 동작이다.

RIO의 `rio_readn()`은 필요한 바이트 수를 다 읽을 때까지 반복해서 `read()`를 호출한다. `rio_writen()`은 필요한 바이트 수를 다 쓸 때까지 반복해서 `write()`를 호출한다. `rio_readlineb()`는 내부 버퍼를 사용해 텍스트 라인을 안정적으로 읽는다. 즉 RIO는 시스템 콜 자체가 아니라, 네트워크 프로그램에서 흔한 부분 읽기/쓰기 문제를 감싸는 견고한 I/O 유틸리티다.

### 17. 앱을 터미널에서 켰을 경우와 GUI 아이콘 클릭으로 켰을 경우 FD table에서 0, 1, 2번 값이 다른 이유? (터미널인 경우 0, 1, 2 모두 /dev/ttys000, GUI인 경우 /dev/null) (정범진)

프로그램을 누가 어떤 환경에서 실행했는지가 다르기 때문이다. 터미널에서 실행하면 셸이 자식 프로세스를 만들면서 0, 1, 2를 보통 같은 터미널 장치(`/dev/ttys000`, `/dev/pts/N` 등)에 연결해 준다. 그래서 키보드 입력을 받고 화면에 출력할 수 있다.

반면 GUI 아이콘으로 실행하면 터미널이 없다. 런처나 데스크톱 환경은 표준 입출력을 `/dev/null`, 로그 파일, 파이프 등으로 연결할 수 있다. 커널이 GUI 프로그램이라서 특별 처리하는 것이 아니라, 부모 프로세스가 자식을 만들 때 어떤 fd를 상속시키느냐의 차이다.

### 18. V-node는 왜 필요한거지? (정범진)

v-node는 여러 파일 시스템을 하나의 공통 모델로 다루기 위한 추상 객체다. 디스크에는 ext4, xfs, tmpfs, procfs, nfs처럼 서로 다른 파일 시스템이 있을 수 있는데, 사용자 프로그램은 모두 `open/read/write/close`로 다루길 원한다.

v-node 또는 리눅스의 inode/dentry/file 구조는 “파일 시스템별 구현 차이”를 VFS 아래로 숨긴다. VFS는 공통 인터페이스를 제공하고, 실제 동작은 각 파일 시스템의 operation table로 위임한다. 그래서 ext4 파일도, procfs 파일도, 네트워크 파일 시스템 파일도 같은 방식으로 열고 읽을 수 있다.

## D-2. 프로세스 족보 & fd 수명

### 1. 모든 프로세스가 fork로 만들어진다면, 그 포크의 “맨 처음” 부모는 누구인가? (최우녕)

리눅스 기준으로 일반 사용자 공간 프로세스의 최상위 조상은 PID 1 프로세스다. 현대 리눅스에서는 보통 `systemd`가 PID 1이다. 커널이 부팅된 뒤 최초의 사용자 공간 프로세스로 PID 1을 실행하고, 이후 대부분의 서비스와 로그인 세션은 그 아래에서 파생된다.

단, PID 1 자체는 일반적인 의미에서 다른 사용자 프로세스가 `fork()`해서 만든 것이 아니다. 커널 부팅 과정에서 생성되고 실행된다. 그래서 “모든 일반 프로세스의 조상은 PID 1이지만, PID 1의 부모는 커널 부팅 과정”이라고 보는 것이 정확하다.

### 2. 터미널에서 프로그램을 띄우면 터미널이 부모인가? 셸(bash)이 부모인가? (최우녕)

보통은 셸이 직접 부모다. 터미널 에뮬레이터는 화면과 키보드 입력을 제공하고, 그 안에서 bash/zsh 같은 셸이 실행된다. 사용자가 `ls`나 `./server`를 입력하면 그 명령을 해석한 셸이 `fork()` 후 `execve()`를 호출한다.

따라서 일반적인 트리는 다음과 같다.

```text
터미널 에뮬레이터
  └── 셸(bash/zsh)
       └── 실행한 프로그램
```

터미널은 조부모에 가깝고, 실제 명령의 부모는 셸이다.

### 3. “크롬이 떠 있는데 엣지를 열면 엣지는 크롬의 자식인가?” 부모가 되는 기준이 뭔가? (최우녕)

아니다. 부모가 되는 기준은 “누가 그 프로세스를 생성했는가”이다. 구체적으로는 어떤 프로세스가 `fork()` 또는 그에 준하는 생성 API를 호출하고, 이어서 `execve()`로 새 프로그램을 실행했는지가 기준이다.

크롬이 이미 떠 있어도, 사용자가 데스크톱 런처에서 엣지를 실행했다면 엣지의 부모는 보통 런처, 윈도우 매니저, 셸, 또는 운영체제의 서비스 프로세스다. 크롬이 직접 엣지를 실행한 경우에만 엣지가 크롬의 자식이 된다.

### 4. 자식이 먼저 죽으면 고아가 된 손자는 “할아버지”로 reparent되는가? (최우녕)

항상 할아버지로 가는 것은 아니다. 리눅스에서는 부모가 죽은 고아 프로세스가 가장 가까운 subreaper에게 입양된다. 그런 subreaper가 없으면 PID 1로 reparent된다.

예를 들어 `A -> B -> C`에서 B가 죽었을 때 C가 반드시 A의 자식이 되는 것은 아니다. A가 subreaper로 등록되어 있으면 A에게 갈 수 있지만, 그렇지 않으면 PID 1 또는 더 가까운 subreaper에게 간다. 이 메커니즘은 고아 프로세스가 종료되었을 때 누군가 `wait()`로 회수할 수 있게 하기 위한 것이다.

### 5. subreaper 플래그는 어디서 언제 켜는가? systemd --user, tini 같은 게 이걸 쓰나? (최우녕)

subreaper는 `prctl(PR_SET_CHILD_SUBREAPER, 1)` 시스템 콜로 현재 프로세스에 설정한다. 이 플래그가 켜진 프로세스는 자기 하위 프로세스 트리에서 고아가 생겼을 때 PID 1 대신 입양자가 될 수 있다.

컨테이너 환경의 `tini`, `dumb-init`, 일부 `systemd --user` 같은 프로세스가 이 역할을 사용할 수 있다. 목적은 자식이나 손자 프로세스가 고아가 되었을 때 좀비로 방치되지 않도록 `wait()`로 회수하는 것이다.

### 6. fork하면 fdtable까지 복제된다는데, 그럼 stdin/stdout/stderr도 공유하는 건가? (최우녕)

fd table의 “슬롯”은 자식에게 복제된다. 그리고 각 슬롯은 보통 부모와 같은 `struct file`을 가리킨다. 따라서 `stdin`, `stdout`, `stderr`에 해당하는 fd 0, 1, 2도 부모와 자식이 같은 열린 파일 객체를 공유하는 경우가 일반적이다.

공유한다는 말은 두 프로세스가 같은 터미널, 같은 파이프, 같은 파일 offset을 볼 수 있다는 뜻이다. 예를 들어 부모와 자식이 같은 파일 fd에 쓰면 출력이 섞일 수 있다. 터미널 stdout을 공유하면 둘 다 같은 터미널에 출력한다.

### 7. 그런데 실제론 어떤 프로세스는 터미널을 물려받고(예: ls), 어떤 건 안 물려받는 것 같다(예: systemd로 띄운 서비스는 터미널에 출력 안 뜸). 기준과 메커니즘이 뭔가? (최우녕)

자식 프로세스가 어떤 fd 0, 1, 2를 상속받았는지의 차이다. 셸에서 실행한 프로그램은 셸이 가진 터미널 fd를 그대로 상속받는다. 그래서 출력이 터미널에 보인다.

반면 systemd 서비스는 터미널에서 직접 실행된 자식이 아니다. systemd가 서비스 프로세스를 만들 때 표준 입출력을 journal, `/dev/null`, 특정 파일, 소켓 등으로 연결할 수 있다. 그래서 서비스가 `printf()`를 해도 사용자의 현재 터미널에 보이지 않는다. 이것은 프로그램의 종류 때문이 아니라 실행한 부모와 fd 설정 때문이라고 보면 된다.

### 8. close(fd) 하면 부모/자식이 공유하던 struct file은 즉시 사라지나? refcount는 언제 0이 되나? (최우녕)

즉시 사라지지 않을 수 있다. `close(fd)`는 현재 프로세스의 fd table 슬롯을 비우고, 그 슬롯이 가리키던 `struct file`의 참조 카운트를 하나 줄인다. 부모와 자식이 같은 `struct file`을 공유하고 있다면 한쪽이 닫아도 다른 쪽 참조가 남아 있으므로 객체는 유지된다.

`struct file`은 참조 카운트가 0이 될 때 실제로 해제된다. 즉 모든 프로세스의 fd table에서 해당 열린 파일 객체를 가리키는 참조가 사라지고, 커널 내부 참조도 없어져야 release 경로가 실행된다.

### 9. O_CLOEXEC는 언제, 누구에게 작용하나? 자식이 fd에 접근하는 순간 닫히는가? (최우녕)

`O_CLOEXEC`는 `execve()`가 성공해 새 프로그램 이미지로 바뀌는 순간 작용한다. 자식이 fd에 접근하는 순간 닫히는 것이 아니다.

흐름은 다음과 같다.

1. 부모가 `open(..., O_CLOEXEC)`로 fd를 연다.
2. `fork()`하면 자식도 그 fd와 close-on-exec 플래그를 물려받는다.
3. 자식이 `execve()`를 호출하기 전에는 그 fd를 읽고 쓸 수 있다.
4. 자식이 `execve()`에 성공하면 커널이 close-on-exec가 켜진 fd들을 닫고 새 프로그램을 시작한다.

이 기능은 비밀 파일, 소켓, 파이프 같은 fd가 의도치 않게 새 프로그램으로 누출되는 것을 막기 위해 사용한다.

### 10. 파일 fd와 소켓 fd가 “완전히 똑같은 규칙”으로 작동한다는 건 어디까지 사실인가? (최우녕)

fd table, `close()`, `dup()`, `fork()` 상속, `O_CLOEXEC`, `poll()` 같은 공통 규칙은 상당히 비슷하게 적용된다. 사용자 입장에서는 둘 다 정수 fd이고 `read/write/close`를 호출할 수 있다.

하지만 실제 의미는 다르다. 일반 파일은 디스크나 page cache의 파일 내용을 offset 기준으로 읽고 쓴다. TCP 소켓은 연결 상태와 송수신 큐를 가진 바이트 스트림이고, UDP 소켓은 메시지 단위 데이터그램을 다룬다. 일반 파일에는 seek가 가능하지만 소켓에는 불가능하다. 따라서 “핸들 관리 규칙은 비슷하지만, 객체별 I/O 의미는 다르다”가 정확하다.

### 11. 소켓 하나의 close()가 TCP는 60초 FIN/ACK, UDP는 즉시 끝 — 같은 함수가 어떻게 다르게 동작하나? (최우녕)

`close(fd)`의 공통 부분은 fd table 참조를 제거하는 것이다. 그 다음 실제 정리는 객체 타입별 release 함수가 처리한다. 일반 파일, TCP 소켓, UDP 소켓은 모두 다른 release 경로를 가진다.

TCP는 연결 지향 프로토콜이므로 `close()` 시 FIN을 보내고 상대의 ACK/FIN을 처리하며 상태 전이를 거친다. 능동 종료한 쪽은 TIME_WAIT 상태에 남을 수 있고, 이 시간이 흔히 약 60초 정도로 보일 수 있다. 반면 UDP는 연결 상태와 FIN/ACK 절차가 없으므로 소켓을 닫으면 커널의 UDP 소켓 객체와 버퍼를 정리하는 것으로 끝난다. 같은 `close()`라도 내부 함수 포인터가 TCP/UDP별 구현으로 분기하기 때문에 동작이 달라진다.

### 12. fork로 복제된 fd를 부모/자식이 동시에 read/write하면 무슨 일이 일어나나? (최우녕)

같은 `struct file`을 공유하면 파일 offset도 공유한다. 따라서 일반 파일에서 부모와 자식이 같은 fd로 `read()`하면 서로 같은 offset을 경쟁적으로 전진시키며 읽는다. 한쪽이 먼저 읽은 바이트는 offset이 이동하므로 다른 쪽은 그 다음 위치를 읽게 된다.

`write()`도 같은 열린 파일 객체를 공유한다. 일반 파일에서는 출력 순서가 스케줄링에 따라 섞일 수 있고, `O_APPEND` 같은 플래그가 있으면 각 write가 파일 끝에 붙도록 커널이 처리한다. 소켓이나 파이프에서는 수신 큐/송신 큐를 공유하므로 여러 프로세스가 동시에 읽으면 먼저 깨어난 쪽이 데이터를 소비하고, 동시에 쓰면 바이트 스트림이 섞일 수 있다. 그래서 동시 접근에는 락, 프로토콜 설계, 프로세스 역할 분리가 필요하다.

### 13. fd로 연결할 수 있는 객체는 뭐가 있나? 전부 나열하고 언제 쓰는지까지 알고 싶다. (최우녕)

대표적으로 다음 객체들이 fd로 연결된다.

- 일반 파일: 디스크 파일 읽기/쓰기
- 디렉터리: 디렉터리 순회, 메타데이터 접근
- 파이프: 부모-자식 또는 관련 프로세스 간 단방향 통신
- FIFO: 이름이 있는 파이프, 무관한 프로세스 간 통신
- TCP/UDP 소켓: 네트워크 통신
- 유닉스 도메인 소켓: 같은 머신 안의 프로세스 간 통신
- 터미널/PTY: 셸, 터미널 입출력
- 문자/블록 디바이스: `/dev/null`, 디스크, 시리얼 장치 등
- procfs/sysfs 파일: 커널 상태 조회와 설정
- eventfd: 커널 이벤트 카운터, 스레드/프로세스 알림
- timerfd: 타이머 이벤트를 fd로 받기
- signalfd: 시그널을 fd read로 받기
- inotify fd: 파일 시스템 변경 감시
- epoll fd: 여러 fd의 이벤트 감시
- pidfd: 프로세스 자체를 안전하게 가리키는 핸들

공통점은 모두 “커널 객체를 사용자 프로세스가 fd라는 정수로 참조한다”는 것이다.

## D-3. 가상 메모리 & 프로세스 레이아웃

### 1. libc는 가상 메모리의 어디에 있는가? 힙 영역인가? 커널 메모리인가? (최우녕)

libc는 커널 메모리가 아니라 사용자 공간 가상 메모리에 매핑된다. 보통 동적 링커가 프로그램 시작 시 libc 공유 라이브러리 파일을 `mmap()`으로 프로세스 주소 공간에 매핑한다.

힙 영역도 아니다. libc의 코드 영역은 읽기/실행 가능한 공유 매핑으로 들어가고, 전역 데이터 영역은 쓰기 가능한 매핑으로 들어간다. 여러 프로세스가 같은 libc 코드 페이지를 물리 메모리에서 공유할 수 있지만, 각 프로세스의 가상 주소 공간에는 자기 것처럼 보인다.

### 2. 가상 메모리 레이아웃 다이어그램의 방향 — 스택이 위에서 아래로, 힙이 아래에서 위로 — 맞는가? (최우녕)

일반적인 설명에서는 맞다. 보통 낮은 주소 쪽에 코드, 데이터, BSS, 힙이 있고, 높은 주소 쪽에 스택이 있다. 힙은 `brk`가 증가하면서 높은 주소 방향으로 자라고, 스택은 함수 호출과 지역 변수 할당에 따라 낮은 주소 방향으로 자란다.

다만 실제 주소 배치는 운영체제, 아키텍처, ASLR 설정, PIE 여부에 따라 달라질 수 있다. “스택은 아래로, 힙은 위로”는 전형적인 개념도에서의 방향이며, 핵심은 두 영역이 서로 다른 방식으로 확장된다는 점이다.

### 3. 힙(brk/sbrk)과 mmap은 뭐가 다른가? (최우녕)

`brk/sbrk` 기반 힙은 프로세스의 전통적인 heap 끝 주소를 움직여 하나의 연속적인 익명 메모리 영역을 키우는 방식이다. 작은 `malloc()` 요청은 보통 이 힙 영역에서 관리된다. 중간 조각을 운영체제에 바로 돌려주기 어렵고, allocator가 내부에서 재사용한다.

`mmap()`은 독립적인 가상 메모리 영역(VMA)을 새로 만드는 방식이다. 파일을 메모리에 매핑할 수도 있고, 파일 없는 익명 메모리를 만들 수도 있다. 큰 `malloc()` 요청은 allocator가 `mmap()`을 사용할 수 있으며, 이 경우 `munmap()`으로 영역 단위 반환이 쉽다. 즉 `brk`는 전통적 연속 힙 확장, `mmap`은 독립 매핑 생성이라고 보면 된다.

### 4. mmap을 호출하면 즉시 물리 메모리가 할당되는가? 힙은? (demand paging의 실체) (최우녕)

대부분의 경우 즉시 물리 메모리가 전부 할당되지 않는다. `mmap()`이나 `brk()`는 먼저 가상 주소 공간에 “이 범위는 사용할 수 있다”는 매핑 정보를 만든다. 실제 물리 페이지는 그 주소를 처음 읽거나 쓸 때 page fault가 발생하면서 할당된다.

이것이 demand paging이다. 운영체제는 요청이 들어왔다고 모든 물리 메모리를 미리 잡아 두지 않고, 실제 접근이 발생한 페이지 단위로 물리 메모리를 연결한다. 파일 매핑이면 필요한 파일 페이지를 읽어 오고, 익명 매핑이면 보통 0으로 초기화된 새 페이지를 연결한다.

### 5. VSZ와 RSS의 차이, overcommit, OOM killer와 이들의 관계는? (최우녕)

VSZ는 프로세스가 예약하거나 매핑한 전체 가상 주소 공간 크기다. 아직 실제 물리 메모리를 쓰지 않는 영역도 포함될 수 있다. RSS는 현재 실제 물리 메모리에 올라와 있는 페이지 크기다.

overcommit은 운영체제가 실제 물리 메모리보다 더 많은 가상 메모리 할당을 허용하는 정책이다. 프로그램이 `malloc(10GB)`를 해도 즉시 10GB 물리 메모리를 쓰지 않을 수 있으므로, 리눅스는 어느 정도 “나중에 실제로 쓰겠지”라고 보고 허용한다.

문제는 많은 프로세스가 예약한 메모리를 실제로 만지기 시작해 물리 메모리와 스왑이 부족해질 때다. 이때 커널은 OOM killer를 실행해 점수가 높은 프로세스를 강제로 종료시켜 시스템 전체가 멈추는 것을 막는다. 따라서 VSZ가 크다고 바로 위험한 것은 아니고, RSS와 실제 메모리 압박이 중요하다.

## D-4. 코드 디테일 — adder.c

### 1. adder.c에서 왜 그냥 printf()로 바로 출력하지 않고, 굳이 sprintf()로 content 문자열을 먼저 만들어야 하나요? 그리고 content에 `=`로 문자열을 넣으면 안 되나요? (CSAPP 3판 920페이지) (이우진)

CGI 응답에서는 본문을 보내기 전에 `Content-length` 헤더를 정확히 계산해야 한다. `printf()`로 본문을 바로 출력해 버리면, 출력하기 전에는 본문 길이를 알기 어렵다. 그래서 먼저 `sprintf(content, ...)`로 본문 문자열을 메모리 버퍼에 만든 뒤 `strlen(content)`로 길이를 계산하고, 그 길이를 헤더에 넣은 다음 헤더와 본문을 순서대로 출력한다.

예를 들어 흐름은 다음과 같다.

```c
sprintf(content, "Welcome to add.com: ");
sprintf(content, "%sTHE Internet addition portal.\r\n<p>", content);
sprintf(content, "%sThe answer is: %d + %d = %d\r\n<p>",
        content, n1, n2, n1 + n2);

printf("Content-length: %d\r\n", (int)strlen(content));
printf("Content-type: text/html\r\n\r\n");
printf("%s", content);
```

`content = "문자열"`처럼 대입할 수 없는 이유는 `content`가 보통 `char content[MAXLINE];` 같은 배열이기 때문이다. C에서 배열 이름은 대입 가능한 변수가 아니다. 배열 안에 문자열을 복사하려면 `strcpy`, `snprintf`, `sprintf` 같은 함수를 써야 한다.

또한 실제 코드에서는 안전성을 위해 `sprintf()`보다 버퍼 크기를 제한할 수 있는 `snprintf()`가 더 좋다. CSAPP 예제는 개념 설명용이라 `sprintf()`를 사용하지만, 실무에서는 버퍼 오버플로우를 피하기 위해 `snprintf(content, sizeof(content), ...)` 형태를 우선 고려한다.
