# Q04. IP 주소 체계 (IPv4 32비트 / IPv6) 와 네트워크 바이트 순서

> CSAPP 11.3 | IP 주소, inet_pton/inet_ntop, htons/ntohs | 기본

## 질문

1. IPv4 는 32비트 정수 한 개라는데, 어떻게 그것만으로 전 세계 주소를 표현하는가.
2. IPv4 와 IPv6 의 주소 형식과 크기는 어떻게 다른가.
3. `htons`, `ntohs`, `htonl`, `ntohl` 은 왜 써야 하는가. 안 쓰면 무슨 일이 벌어지는가.

## 답변

### 최우녕

> IPv4 는 32비트 정수 한 개라는데, 어떻게 그것만으로 전 세계 주소를 표현하는가.

IPv4 주소는 `uint32_t` 한 개, 즉 $2^{32} \approx 42.9\text{억}$ 개의 주소가 존재한다. 이걸 사람이 읽기 편하게 1바이트씩 끊어 `a.b.c.d` 로 표기한 것이 dotted-decimal notation이다.

"어떻게 전 세계를 커버하느냐"에 대한 답은 세 가지 요소가 합쳐진 결과다.

첫째, **계층적 할당**. IANA가 ISP 또는 RIR(지역 인터넷 레지스트리)에게 큰 블록을 주고, RIR 이 사업자에게, 사업자가 조직에 주는 식이다. 각 조직은 받은 블록(예: `128.2.0.0/16` = 65,536개 주소)을 내부에서 서브넷으로 나눠 쓴다. 라우팅은 **주소가 아니라 "블록(prefix) 단위"** 로 이뤄지므로 라우팅 테이블은 훨씬 작다.

둘째, **CIDR(Classless Inter-Domain Routing)**. 예전에는 A/B/C 클래스로 크기가 고정이었으나, 지금은 `/24`, `/16`, `/29` 처럼 **프리픽스 길이를 자유롭게 지정**한다. 예: `10.0.0.0/8` 은 앞 8비트가 동일한 주소 모두를 의미한다.

```text
128.2.194.242  ->  이진수로
10000000.00000010.11000010.11110010

/16 프리픽스면:  128.2.0.0/16 (앞 16비트가 같은 65,536개 주소)
/24 프리픽스면:  128.2.194.0/24 (앞 24비트가 같은 256개 주소)
```

셋째, **NAT(Network Address Translation)**. 42.9억 개는 세계 디바이스 수(수백억)에 이미 부족하다. 실제로는 집/회사에서 **사설 주소(10.x, 172.16~31.x, 192.168.x)** 를 쓰고, 게이트웨이가 나갈 때만 공인 IP 하나로 바꿔 준다. 안쪽 프로세스 수십만 개가 **공인 IP 1개 + 서로 다른 포트**로 매핑된다.

> IPv4 와 IPv6 의 주소 형식과 크기는 어떻게 다른가.

```text
IPv4
  크기     : 32 비트 = 4 바이트
  형식     : a.b.c.d  (각 0~255)
  공간     : 약 4.29 × 10^9
  구조체   : struct in_addr  { uint32_t s_addr; }

IPv6
  크기     : 128 비트 = 16 바이트
  형식     : 8개의 16비트 hextet 을 ":" 로 연결
             예) 2001:0db8:85a3:0000:0000:8a2e:0370:7334
             연속된 0 은 "::" 로 한 번만 축약 가능
             -> 2001:db8:85a3::8a2e:370:7334
  공간     : 약 3.4 × 10^38 (지구 표면 cm² 당 수조 개)
  구조체   : struct in6_addr { uint8_t s6_addr[16]; }
```

실무적인 차이 몇 가지도 같이 기억해두면 좋다:

- IPv6 에는 브로드캐스트가 없고 멀티캐스트로 대체되었다.
- IPv6 헤더는 고정 40B 로 단순화되었다(옵션은 확장 헤더로 분리).
- IPv6 는 NAT 없이도 주소가 충분하므로, 원칙적으로 호스트가 직접 공인 주소를 가질 수 있다.
- CSAPP 의 `getaddrinfo` 설계는 이 두 버전을 모두 다루기 위한 프로토콜 독립적 인터페이스다(= `sockaddr_storage`, `ai_family = AF_INET | AF_INET6 | AF_UNSPEC`).

> `htons`, `ntohs`, `htonl`, `ntohl` 은 왜 써야 하는가. 안 쓰면 무슨 일이 벌어지는가.

네트워크는 **빅 엔디안(big-endian)** 을 표준 바이트 순서로 쓴다. 반면 x86/ARM(Apple silicon 포함) 계열 CPU 의 메모리 표현은 **리틀 엔디안(little-endian)** 이다. 이 차이를 맞춰주는 게 이 네 함수다.

- `htons` : host -> network short(16-bit) — 주로 포트 번호 변환
- `htonl` : host -> network long(32-bit) — 주로 IPv4 주소 변환
- `ntohs` : network -> host short
- `ntohl` : network -> host long

예를 보자. 포트 80(`0x0050`)을 리틀엔디안 x86 메모리에 그냥 쓰면:

```text
uint16_t port = 80;            // 0x0050
메모리 바이트 배열(LE):         [0x50, 0x00]

네트워크 상에서 80 을 쓰려면 BE 로:   [0x00, 0x50]
htons(80) = 0x5000             // BE 표현을 LE 레지스터에 담으면 이렇게 보인다

sin_port = htons(80);          // 소켓 구조체엔 이대로 들어가야 한다
```

만약 `htons` 없이 `sin_port = 80` 으로 직접 넣으면 어떻게 되는가?

```text
sin_port 의 바이트들(LE 메모리):  [0x50, 0x00]
네트워크로 그대로 나가면 상대편은 BE 로 해석 -> 0x5000 = 20480
=> 서버는 20480 포트로 연결 시도한 것으로 이해 -> 전혀 엉뚱한 포트!
```

즉 포트/주소를 소켓 구조체에 넣을 때는 **반드시 host order -> network order 변환**이 필요하다. 반대로 패킷에서 읽어온 값을 비교하거나 출력할 때는 network order -> host order (`ntohs`, `ntohl`) 로 돌려 써야 사람이 기대하는 값이 된다.

64비트 변환 함수가 CSAPP에서 언급되지 않는 이유는 표준에 없기 때문이다 (비표준 `htonll`을 쓰거나 직접 구현). IP/포트는 최대 32비트라 `htonl`/`htons` 까지만 있으면 충분하다.

IP 주소 변환은 문자열<->바이너리 변환이므로 별도다:

- `inet_pton(AF_INET, "128.2.194.242", &dst)` -> `struct in_addr` 에 BE 로 채움
- `inet_ntop(AF_INET, &src, buf, INET_ADDRSTRLEN)` -> 사람이 읽는 문자열로
