# 05. CSAPP 11 전체 순차 수치 워크스루

이 문서는 `CSAPP` 11장을 네트워크 초보자 기준으로 다시 풀어쓴 **학습 기준서**입니다.

목표는 단순히 목차를 따라가는 것이 아니라, 아래 질문에 끝까지 답할 수 있게 만드는 것입니다.

- 브라우저가 `http://www.example.net/home.html`을 입력하면 실제로 어떤 숫자와 구조가 오가는가
- `IP`, `포트`, `소켓`, `HTTP`, `Tiny`가 한 줄로 어떻게 연결되는가
- `Tiny -> Proxy -> SQL API 서버`로 확장할 때 무엇이 재사용되는가

이 문서는 책의 11.1 -> 11.7 흐름을 그대로 따르되, 각 절마다 아래 네 가지를 반드시 남깁니다.

- 한 줄 정의
- 왜 필요한가
- 실제 숫자를 넣은 예시
- 초보자가 자주 헷갈리는 포인트

주의:

- 아래 IP, MAC, 포트, 바이트 수 일부는 **이해를 위한 예시 값**입니다
- 개념 흐름은 `CSAPP` 11장을 따르되, 설명을 더 명확하게 하기 위해 보충 설명을 덧붙였습니다
- 이 문서는 `CSAPP 11장 범위`를 완성도 있게 이해하기 위한 문서이지, 네트워크 전 영역을 끝내는 문서는 아닙니다

## 0. 끝까지 사용할 공통 시나리오

이 문서 전체는 아래 한 개의 시나리오를 계속 확장합니다.

### 등장인물

| 역할 | 값 |
| --- | --- |
| 클라이언트 호스트 A | `128.2.194.242` |
| 클라이언트 임시 포트 | `51213` |
| 클라이언트 MAC | `AA:AA:AA:AA:AA:AA` |
| 라우터 R의 LAN1 MAC | `11:11:11:11:11:11` |
| 라우터 R의 LAN2 MAC | `22:22:22:22:22:22` |
| 웹 서버 호스트 B | `208.216.181.15` |
| 웹 서버 포트 | `80` |
| 웹 서버 MAC | `BB:BB:BB:BB:BB:BB` |
| 로컬 Tiny 서버 예시 포트 | `15213` |

### 사람이 보는 요청

사람은 보통 아래처럼 URL을 봅니다.

```text
http://www.example.net/home.html
```

브라우저와 커널은 결국 아래 정보가 필요합니다.

- host: `www.example.net`
- IP: `208.216.181.15`
- port: `80`
- path: `/home.html`

### HTTP 요청 예시

```text
GET /home.html HTTP/1.0\r\n
Host: www.example.net\r\n
Connection: close\r\n
Proxy-Connection: close\r\n
\r\n
```

줄 길이를 대략 계산하면:

- 요청 줄 `GET /home.html HTTP/1.0\r\n` -> 약 `25B`
- `Host: www.example.net\r\n` -> 약 `24B`
- `Connection: close\r\n` -> 약 `19B`
- `Proxy-Connection: close\r\n` -> 약 `25B`
- 빈 줄 `\r\n` -> `2B`

총 요청 헤더 크기 예시:

```text
25 + 24 + 19 + 25 + 2 = 95B
```

### 정적 응답 예시

서버가 `home.html`을 정적으로 준다고 가정합니다.

- 파일 크기: `2048B`
- 응답 헤더 크기 예시: `91B`
- 전체 응답 크기 예시: `2139B`

```text
91B + 2048B = 2139B
```

### 동적 응답 예시

서버가 CGI 프로그램 `adder`를 실행한다고 가정합니다.

```text
GET /cgi-bin/adder?15000&213 HTTP/1.0
```

이때 프로그램은 아래 계산을 수행합니다.

```text
15000 + 213 = 15213
```

즉 이 문서에서는 정적 요청과 동적 요청을 모두 사용합니다.

## 1. 30초로 보는 11장 전체

브라우저에 URL을 입력한 뒤 화면이 뜨기까지를 30초 요약으로 적으면 아래와 같습니다.

1. 사람이 `http://www.example.net/home.html`을 입력합니다.
2. DNS가 `www.example.net -> 208.216.181.15`를 찾습니다.
3. 클라이언트는 커널에게 서버 `208.216.181.15:80`에 연결해 달라고 요청합니다.
4. 커널은 임시 포트 `51213`을 골라 연결을 만듭니다.
5. 이 연결은 `(128.2.194.242:51213, 208.216.181.15:80)`로 식별됩니다.
6. 브라우저는 `95B`짜리 HTTP 요청을 보냅니다.
7. 이 요청은 TCP, IP, Ethernet 헤더가 붙어 약 `153B`짜리 프레임으로 LAN을 흐릅니다.
8. 라우터는 IP는 유지하고 MAC만 바꿔 다음 LAN으로 전달합니다.
9. 서버는 `accept`로 연결을 받고, Tiny는 요청 줄과 헤더를 읽고, 정적이면 파일을 보내고 동적이면 CGI를 실행합니다.
10. 클라이언트는 응답을 읽고 HTML을 렌더링합니다.

이 한 줄이 11장 전체입니다.

```mermaid
flowchart LR
    A["URL 입력"] --> B["DNS: 이름 -> IP"]
    B --> C["connect: 서버 80 연결"]
    C --> D["socket pair 생성"]
    D --> E["HTTP 요청 95B 전송"]
    E --> F["TCP/IP/Ethernet 캡슐화"]
    F --> G["라우터 전달"]
    G --> H["accept: 서버 연결 수락"]
    H --> I["Tiny가 URI 파싱"]
    I --> J["static 파일 또는 CGI 실행"]
    J --> K["HTTP 응답 전송"]
    K --> L["브라우저 렌더링"]
```

## 11.1 The Client-Server Programming Model

### 한 줄 정의

네트워크 프로그램은 결국 **클라이언트가 요청하고 서버가 자원을 제공하는 트랜잭션**입니다.

### 왜 필요한가

초보자가 가장 먼저 잡아야 하는 기준은 "네트워크 서버도 결국 요청 -> 처리 -> 응답"이라는 점입니다.

이 기준이 잡혀야:

- 웹 서버를 이해할 수 있고
- 프록시를 이해할 수 있고
- SQL API 서버도 같은 틀에서 볼 수 있습니다

### 4단계 트랜잭션을 숫자로 보기

#### 1. 클라이언트가 요청을 보낸다

- 목적지 서버: `208.216.181.15:80`
- 요청 크기: `95B`

#### 2. 서버가 요청을 해석한다

서버는 요청 줄을 읽고:

- 메서드: `GET`
- 자원: `/home.html`
- 버전: `HTTP/1.0`

를 해석합니다.

#### 3. 서버가 응답을 만든다

정적 파일이라고 가정하면:

- 파일 크기: `2048B`
- 응답 헤더: `91B`
- 전체 응답: `2139B`

#### 4. 클라이언트가 응답을 소비한다

브라우저는 `2139B`를 읽고 HTML을 렌더링합니다.

### 이 절에서 꼭 잡아야 할 감각

- 서버는 클라이언트 대신 자원을 보관하고 관리합니다
- 클라이언트는 요청을 통해서만 자원에 접근합니다
- "파일을 주는 서버", "프록시 서버", "SQL API 서버"는 자원만 다를 뿐 구조는 같습니다

### 초보자 오해 포인트

- 서버는 항상 파일만 주는 프로그램이 아닙니다
- 응답은 꼭 HTML일 필요가 없습니다
- "네트워크 프로그램"은 특별한 종류의 프로그램이 아니라, I/O가 네트워크를 통해 일어나는 프로그램입니다

### 이 절을 이해했다는 뜻

아래 질문에 답할 수 있으면 됩니다.

- 서버가 관리하는 자원은 무엇인가
- 클라이언트는 왜 직접 서버 디스크를 읽지 못하는가
- SQL API 서버도 왜 client-server 모델인지

## 11.2 Networks

### 한 줄 정의

네트워크는 여러 LAN과 라우터가 연결되어 데이터를 단계적으로 전달하는 구조입니다.

### 왜 필요한가

11.1에서 "요청과 응답"을 봤다면, 11.2에서는 그 요청이 **실제로 어떤 단위로 움직이는지** 알아야 합니다.

사람은 "95바이트를 보냈다"고 생각하지만, 네트워크 장비는 다른 단위로 봅니다.

### application data와 실제 전송 크기

HTTP 요청 `95B`가 전선에 그대로 실리는 것은 아닙니다.

예시:

- HTTP payload: `95B`
- TCP header: `20B`
- IP header: `20B`
- Ethernet header: `14B`
- Ethernet trailer(FCS): `4B`

전선 위 프레임 크기 예시:

```text
95 + 20 + 20 + 14 + 4 = 153B
```

즉 사용자는 `95B`를 보냈다고 생각하지만, LAN에서는 `153B`짜리 Ethernet frame이 움직입니다.

### packet과 frame 차이

이 둘은 초보자가 가장 많이 헷갈리는 용어입니다.

| 용어 | 어느 관점인가 | 무엇이 들어 있는가 |
| --- | --- | --- |
| packet | IP 관점 | 출발지 IP, 목적지 IP, 상위 계층 데이터 |
| frame | LAN 관점 | 출발지 MAC, 목적지 MAC, packet |

우리 예시에서:

- packet 안의 출발지 IP: `128.2.194.242`
- packet 안의 목적지 IP: `208.216.181.15`
- LAN1 frame의 목적지 MAC: `11:11:11:11:11:11`

즉 **IP는 최종 목적지를 가리키고, MAC은 현재 LAN에서 다음으로 넘길 대상을 가리킵니다.**

### 라우터를 지나갈 때 무엇이 바뀌는가

LAN1에서 호스트 A가 라우터 R로 보낼 때:

```text
출발지 MAC = AA:AA:AA:AA:AA:AA
목적지 MAC = 11:11:11:11:11:11
출발지 IP  = 128.2.194.242
목적지 IP  = 208.216.181.15
```

라우터를 지나 LAN2로 나갈 때:

```text
출발지 MAC = 22:22:22:22:22:22
목적지 MAC = BB:BB:BB:BB:BB:BB
출발지 IP  = 128.2.194.242
목적지 IP  = 208.216.181.15
```

정리:

- 바뀌는 것: MAC 주소, Ethernet header/trailer
- 유지되는 것: IP 주소, TCP 포트, HTTP 데이터

### Figure 11.7 감각으로 보는 8단계 전달

우리 예시를 책의 전형적인 흐름으로 풀면 아래처럼 볼 수 있습니다.

1. 브라우저가 `95B` HTTP 요청을 만듭니다.
2. OS가 TCP 헤더 `20B`를 붙입니다.
3. OS가 IP 헤더 `20B`를 붙여 packet을 만듭니다.
4. NIC가 Ethernet 헤더와 trailer를 붙여 LAN1 frame을 만듭니다.
5. frame이 LAN1을 타고 라우터 R까지 갑니다.
6. 라우터는 Ethernet 부분을 벗기고 IP 목적지를 보고 다음 홉을 결정합니다.
7. 라우터는 LAN2용 Ethernet header를 새로 붙입니다.
8. 서버 NIC가 frame을 받고 상위 계층으로 올립니다.

### hub, bridge, router를 초보자 언어로 구분

- hub: 들어온 신호를 퍼뜨리는 단순 장비
- bridge: LAN을 분리하면서 frame을 중계하는 장비
- router: IP 목적지를 보고 서로 다른 네트워크 사이를 연결하는 장비

이 장에서 가장 중요한 것은 **인터넷 전체는 router가 이어 주고, 각 LAN 안에서는 frame이 흐른다**는 점입니다.

### 초보자 오해 포인트

- "IP만 있으면 되지 MAC은 왜 있지?" -> IP는 인터넷 전체의 주소, MAC은 현재 LAN에서 실제 전달 대상입니다
- "라우터가 지나가면 IP도 바뀌나?" -> NAT 같은 별도 장치가 없다고 가정하면 일반적인 라우팅에서는 MAC만 바뀌고 IP는 유지됩니다
- "packet이랑 frame은 같은 말인가?" -> 아닙니다. 보는 계층이 다릅니다

### 이 절을 이해했다는 뜻

- `95B` HTTP 요청이 왜 LAN에서는 `153B` 정도의 frame으로 보이는지 설명할 수 있습니다
- 라우터를 지날 때 MAC이 바뀌고 IP는 유지된다는 것을 설명할 수 있습니다

## 11.3 The Global IP Internet

### 한 줄 정의

서로 다른 네트워크를 하나의 거대한 인터넷처럼 묶는 공통 규칙이 `IP 인터넷`입니다.

### 왜 필요한가

11.2가 "데이터가 움직이는 방식"이라면, 11.3은 "어디로 가야 하는지 어떻게 식별하는가"를 설명합니다.

핵심은 세 가지입니다.

- IP 주소
- 도메인 이름
- 연결을 식별하는 socket pair

## 11.3.1 IP Addresses

### 한 줄 정의

IP 주소는 인터넷에서 호스트를 식별하기 위한 **32비트 숫자**입니다.

### dotted decimal과 실제 숫자

예시 IP:

```text
128.2.194.242
```

각 옥텟을 16진수로 바꾸면:

- `128` -> `0x80`
- `2` -> `0x02`
- `194` -> `0xC2`
- `242` -> `0xF2`

따라서 32비트 값은:

```text
0x80 02 C2 F2
= 0x8002C2F2
```

즉 `128.2.194.242`는 사람이 읽기 좋게 점으로 끊어 쓴 것이고, 실제로는 32비트 숫자입니다.

### network byte order

TCP/IP는 big-endian을 network byte order로 씁니다.

즉 네트워크로 나갈 때 바이트 순서는:

```text
80 02 C2 F2
```

만약 CPU가 little-endian이면 메모리 안에서는 다르게 저장될 수 있습니다.
그래서 아래 함수가 필요합니다.

- `htonl`: host to network long
- `htons`: host to network short
- `ntohl`: network to host long
- `ntohs`: network to host short

### 숫자로 보는 포트

웹 포트 `80`은:

```text
80(decimal) = 0x0050
```

Tiny 포트 `15213`은:

```text
15213(decimal) = 0x3B6D
```

네트워크 바이트 순서에서는:

```text
3B 6D
```

로 나갑니다.

### 사람이 읽는 문자열과 숫자 변환

`CSAPP` 11장에서는 아래 함수도 중요합니다.

- `inet_pton`: 사람이 읽는 IP 문자열 -> 네트워크용 이진 표현
- `inet_ntop`: 이진 표현 -> 사람이 읽는 문자열

예시 감각:

```c
inet_pton(AF_INET, "208.216.181.15", &addr.sin_addr);
```

### localhost도 꼭 기억

책은 `127.0.0.1`도 언급합니다.

- `localhost`
- `127.0.0.1`

는 "이 컴퓨터 자기 자신"을 가리키는 대표적인 이름/주소입니다.

### 초보자 오해 포인트

- IP 주소는 문자열이 아니라 본질적으로 숫자입니다
- 포트도 문자열이 아니라 숫자입니다
- `hton*` 계열 함수는 "암호화"가 아니라 바이트 순서를 맞추는 작업입니다

### 이 절을 이해했다는 뜻

- `128.2.194.242`가 왜 32비트 숫자인지 설명할 수 있습니다
- `15213` 포트를 왜 `htons(15213)`로 넣는지 설명할 수 있습니다

## 11.3.2 Internet Domain Names

### 한 줄 정의

도메인 이름은 사람이 IP 숫자 대신 사용하기 위한 계층형 이름입니다.

### 왜 필요한가

사람은 `208.216.181.15`보다 `www.example.net`을 기억하기 쉽습니다.

DNS는 이 둘을 연결합니다.

### 이름 계층 구조

`www.example.net`을 오른쪽부터 보면:

- `net`
- `example`
- `www`

즉 도메인 이름은 계층 구조입니다.

### 숫자로 보는 DNS 질의

브라우저가 이름만 알고 IP를 모른다면 먼저 DNS 질의를 해야 합니다.

예시:

- 질의 대상: 로컬 DNS 서버
- DNS 서비스 포트: `53`
- 결과: `www.example.net -> 208.216.181.15`

즉 실제 웹 연결 전에 이미 "이름을 숫자로 바꾸는 네트워크 왕복"이 선행될 수 있습니다.

### 여러 이름과 여러 주소

초보자는 "이름 하나 = IP 하나"라고 생각하기 쉽지만 그렇지 않습니다.

- 여러 도메인 이름이 하나의 IP를 가리킬 수 있습니다
- 하나의 도메인 이름이 여러 IP를 가질 수도 있습니다

이 점은 나중에 로드밸런싱, 가상 호스팅, CDN을 이해할 때 중요합니다.

### 초보자 오해 포인트

- DNS는 웹 서버가 아닙니다. 이름을 주소로 바꾸는 시스템입니다
- 브라우저는 보통 도메인 이름으로 연결하는 것이 아니라, DNS 결과로 얻은 IP로 연결합니다

### 이 절을 이해했다는 뜻

- 왜 이름과 IP를 분리하는지 설명할 수 있습니다
- DNS 질의가 웹 요청보다 먼저 일어날 수 있다는 것을 설명할 수 있습니다

## 11.3.3 Internet Connections

### 한 줄 정의

인터넷 연결은 두 개의 socket address 쌍으로 식별되는, 신뢰 가능한 전이중 바이트 스트림입니다.

### socket address

socket address는:

```text
IP 주소 + 포트 번호
```

입니다.

예시:

- 클라이언트 socket address: `128.2.194.242:51213`
- 서버 socket address: `208.216.181.15:80`

연결 전체는 socket pair로 표현하면:

```text
(128.2.194.242:51213, 208.216.181.15:80)
```

### 왜 포트가 필요한가

IP 주소만으로는 "그 컴퓨터 안의 어떤 프로그램"과 통신해야 하는지 모릅니다.

예시:

- `208.216.181.15:80` -> 웹 서버
- `208.216.181.15:22` -> SSH 서버
- `208.216.181.15:3306` -> 데이터베이스 서버

즉 포트는 한 호스트 안에서 서비스 창구를 구분합니다.

### 왜 임시 포트가 필요한가

클라이언트도 자기 쪽 식별자가 필요합니다.

예시:

```text
(128.2.194.242:51213, 208.216.181.15:80)
(128.2.194.242:51214, 208.216.181.15:80)
```

위 두 연결은 서로 다른 연결입니다.

즉 같은 브라우저가 같은 서버에 두 번 접속해도, 임시 포트가 달라지면 커널은 다른 연결로 구분할 수 있습니다.

### point-to-point, full-duplex, reliable

책이 강조하는 연결의 성질은 아래 세 가지입니다.

- point-to-point: 연결 양 끝이 분명합니다
- full-duplex: 양방향으로 동시에 보낼 수 있습니다
- reliable: 보내는 순서와 무결성을 보장하는 전송을 제공합니다

### 초보자 오해 포인트

- 포트는 파일 번호가 아니라 네트워크 서비스 식별자입니다
- 연결은 서버 포트 하나만으로 식별되지 않습니다. 양쪽 주소와 포트가 모두 필요합니다

### 이 절을 이해했다는 뜻

- 왜 socket pair가 필요한지 설명할 수 있습니다
- 왜 `80` 하나만으로는 연결을 구분할 수 없는지 설명할 수 있습니다

## 11.4 The Sockets Interface

### 한 줄 정의

소켓 인터페이스는 네트워크 연결을 파일 디스크립터처럼 다루게 해 주는 운영체제 API입니다.

### 왜 중요한가

이 장의 실전은 여기부터입니다.

앞 절에서 배운:

- IP 주소
- 포트
- 연결

을 실제 코드로 만드는 API가 `socket`, `connect`, `bind`, `listen`, `accept`입니다.

## 11.4.1 Socket Address Structures

### 한 줄 정의

소켓 주소 구조체는 "어떤 주소 체계, 어떤 포트, 어떤 IP"를 메모리 안에 담는 형식입니다.

### 대표 구조체

IPv4에서는 보통 `struct sockaddr_in`을 씁니다.

```c
struct sockaddr_in addr;
addr.sin_family = AF_INET;
addr.sin_port = htons(15213);
addr.sin_addr.s_addr = htonl(INADDR_ANY);
```

숫자로 보면:

- `AF_INET` -> IPv4 주소 체계
- `15213` -> `0x3B6D`
- `INADDR_ANY` -> `0.0.0.0`

즉:

```text
0.0.0.0:15213
```

은 "이 머신의 어떤 로컬 IP로 들어오든 15213 포트를 받겠다"는 뜻입니다.

### 메모리 감각

`sockaddr_in`은 보통 아래 정보를 담습니다.

- 주소 체계
- 포트
- IPv4 주소
- 패딩

초보자에게 중요한 것은 **소켓 함수가 결국 이 구조체 안에 들어 있는 숫자를 읽는다**는 점입니다.

### 왜 `sockaddr *`로 캐스팅하는가

많은 소켓 함수는 범용 인터페이스를 위해 `struct sockaddr *`를 받습니다.

즉 실제로는 `sockaddr_in`을 준비하고, 호출할 때는 상위 타입처럼 넘기는 경우가 많습니다.

### 초보자 오해 포인트

- `INADDR_ANY`는 "인터넷 전체 아무나 허용"이 아니라 "이 호스트의 어떤 로컬 인터페이스 주소로 와도 받음"에 가깝습니다
- 구조체에 숫자를 넣을 때 포트와 주소는 network byte order를 맞춰야 합니다

### 이 절을 이해했다는 뜻

- 왜 `sin_port = htons(15213)`인지 설명할 수 있습니다
- 왜 서버 bind 예시가 `0.0.0.0:15213`이 되는지 설명할 수 있습니다

## 11.4.2 The socket Function

### 한 줄 정의

`socket`은 "이제 네트워크 통신에 쓸 커널 객체를 하나 만들겠다"는 호출입니다.

### 예시

```c
listenfd = socket(AF_INET, SOCK_STREAM, 0);
```

성공하면 예를 들어:

```text
listenfd = 3
```

을 반환할 수 있습니다.

여기서 핵심은:

- `3`은 정수 file descriptor입니다
- 아직 누구와도 연결된 상태는 아닙니다
- 단지 TCP 소켓 객체가 하나 생긴 것입니다

### 왜 file descriptor 감각이 중요한가

`CSAPP`가 네트워크를 I/O 장치처럼 설명하는 이유가 여기 있습니다.

네트워크 소켓도 결국:

- `read`
- `write`
- `close`

감각으로 다룰 수 있습니다.

### 초보자 오해 포인트

- `socket()` 성공이 곧 연결 성공은 아닙니다
- 이 시점의 fd는 "연결된 상대"가 아니라 "연결에 사용할 수 있는 소켓 객체"입니다

## 11.4.3 The connect Function

### 한 줄 정의

`connect`는 클라이언트가 특정 서버 주소로 실제 연결을 시작하는 호출입니다.

### 예시

```c
connect(clientfd, (SA *)&serveraddr, sizeof(serveraddr));
```

우리 예시로 풀면:

- `clientfd = 3`
- 서버 주소 = `208.216.181.15:80`

성공하면 커널은:

1. 클라이언트 쪽 임시 포트 하나를 고릅니다
2. 예를 들어 `51213`을 붙입니다
3. 서버와 연결을 확립합니다

결과 연결:

```text
(128.2.194.242:51213, 208.216.181.15:80)
```

### 코드 바깥에서 실제로 무슨 일이 일어나는가

책은 상세한 TCP 내부를 이 장에서 깊게 파지 않지만, 초보자 감각으로는 아래 정도를 붙여두면 좋습니다.

- `connect`는 커널에게 "저 서버로 TCP 연결을 만들어 달라"고 부탁하는 호출입니다
- 성공 후에는 그 fd에 `write`하면 서버로 바이트가 갑니다
- 실패하면 주소가 틀렸거나, 서버가 안 듣고 있거나, 네트워크가 닿지 않을 수 있습니다

### 초보자 오해 포인트

- 임시 포트는 보통 개발자가 직접 고르지 않습니다
- 클라이언트도 자신의 포트를 가집니다

## 11.4.4 The bind Function

### 한 줄 정의

`bind`는 소켓에 로컬 주소와 포트를 붙이는 호출입니다.

### 예시

```c
bind(listenfd, (SA *)&serveraddr, sizeof(serveraddr));
```

서버 예시:

```text
listenfd = 3
주소     = 0.0.0.0:15213
```

의미:

- "15213 포트 요청은 내가 받겠다"

입니다.

### 왜 서버에 필요하고 클라이언트는 보통 안 하는가

서버는 모두가 찾아올 고정 포트가 필요합니다.

클라이언트는 보통 커널이 임시 포트를 자동으로 골라 주므로, 일반적인 경우 `connect`만 쓰고 `bind`를 직접 하지 않습니다.

### 초보자 오해 포인트

- `bind`는 데이터를 보내는 호출이 아닙니다
- `bind`를 했다고 아직 손님을 받는 상태는 아닙니다

## 11.4.5 The listen Function

### 한 줄 정의

`listen`은 bound socket을 수동적(passive) listening socket으로 바꾸는 호출입니다.

### 예시

```c
listen(listenfd, 1024);
```

숫자로 보면:

- `1024`는 backlog 예시
- 아직 `accept`되지 않은 연결 요청이 잠시 대기할 수 있는 큐의 크기 개념입니다

이제 `fd=3`은 "문지기"가 됩니다.

중요:

- 아직 특정 클라이언트와 대화하는 fd는 아닙니다
- 연결 요청을 받아들이기 위한 listening descriptor입니다

### 초보자 오해 포인트

- `listenfd`는 요청 데이터를 읽는 채널이 아닙니다
- 실제 대화는 나중에 `accept`가 만들어 주는 별도 `connfd`에서 일어납니다

## 11.4.6 The accept Function

### 한 줄 정의

`accept`는 대기 중이던 연결 요청 하나를 받아, 그 클라이언트 전용 연결 fd를 새로 만드는 호출입니다.

### 예시

```c
connfd = accept(listenfd, (SA *)&clientaddr, &clientlen);
```

예시 결과:

- `listenfd = 3`
- 새 `connfd = 4`
- 클라이언트 주소 = `128.2.194.242:51213`

즉 역할이 나뉩니다.

- `fd 3` -> 계속 문지기
- `fd 4` -> 이번 손님과 실제 대화하는 채널

### 두 손님이 연달아 오면

첫 번째 손님:

```text
accept -> connfd = 4
```

두 번째 손님:

```text
accept -> connfd = 5
```

즉 서버는 `listenfd` 하나를 유지하면서, 손님마다 별도 `connfd`를 만듭니다.

### 왜 이 구분이 중요한가

이 구분이 있어야:

- 순차 서버
- 멀티프로세스 서버
- 멀티스레드 서버
- 스레드풀 서버

모두 확장할 수 있습니다.

### 초보자 오해 포인트

- `listenfd`와 `connfd`를 같은 것으로 보면 서버 구조를 이해하기 어렵습니다
- `accept`가 반환하는 fd가 실제 요청과 응답이 오가는 채널입니다

## 11.4.7 Host and Service Conversion

### 한 줄 정의

`getaddrinfo`와 `getnameinfo`는 이름과 숫자 주소를 안전하게 바꾸기 위한 현대식 인터페이스입니다.

### 클라이언트 예시

```c
getaddrinfo("www.example.net", "80", ...);
```

의미:

- host 이름 `www.example.net`
- service 문자열 `80`

를 실제 연결 가능한 주소 후보 리스트로 바꾸어 달라

결과 예시:

- 후보 1 -> 실패
- 후보 2 -> 실패
- 후보 3 -> `208.216.181.15:80` 성공

### 서버 예시

```c
getaddrinfo(NULL, "15213", ...);
```

의미:

- 내 로컬 인터페이스들에 대해
- `15213` 포트로 listen 가능한 주소 후보를 달라

### 왜 좋은가

- hostname과 service를 문자열로 다룰 수 있습니다
- IPv4/IPv6 독립적인 인터페이스입니다
- 오래된 함수보다 현대적이고 재진입성 측면에서 안전합니다

### `getnameinfo` 감각

서버가 `Accept` 후 클라이언트 주소를 받았을 때:

- `128.2.194.242`
- `51213`

를 사람이 읽는 문자열로 바꾸는 데 씁니다.

이게 Tiny 예제의 접속 로그 출력 감각과 연결됩니다.

## 11.4.8 Helper Functions for the Sockets Interface

### 한 줄 정의

`CSAPP`는 반복되는 소켓 설정 코드를 `open_clientfd`, `open_listenfd`로 감싸서 실수 가능성을 줄입니다.

### open_clientfd 흐름

1. `getaddrinfo("www.example.net", "80")`
2. 후보 주소 리스트를 순회
3. 각 후보마다 `socket`
4. `connect` 시도
5. 성공한 fd 반환

예시:

```text
후보 1: socket 성공, connect 실패
후보 2: socket 성공, connect 실패
후보 3: socket 성공, connect 성공 -> clientfd = 3
```

### open_listenfd 흐름

1. `getaddrinfo(NULL, "15213")`
2. 후보 주소 리스트를 순회
3. 각 후보마다 `socket`
4. `setsockopt(SO_REUSEADDR)`
5. `bind`
6. `listen`
7. 성공한 fd 반환

예시:

```text
socket -> 3
setsockopt(3, SO_REUSEADDR)
bind(3, 0.0.0.0:15213) 성공
listen(3, LISTENQ) 성공
반환 listenfd = 3
```

### 왜 helper가 중요한가

소켓 초기화 코드는:

- 에러 처리 많고
- 후보 주소 반복 많고
- 플랫폼 차이도 신경 써야 하므로

초보자는 핵심 개념보다 보일러플레이트에 묻히기 쉽습니다.

`CSAPP`는 helper로 그 부담을 줄입니다.

## 11.4.9 Example Echo Client and Server

### 한 줄 정의

Echo 예제는 "소켓도 결국 읽고 쓰는 바이트 스트림"이라는 사실을 가장 단순하게 보여 주는 최소 서버입니다.

### 숫자로 보는 Echo

클라이언트가 아래 데이터를 보낸다고 합시다.

```text
PING\n
```

길이:

```text
5B
```

서버는:

1. `read(connfd, buf, ...)`로 `5B`를 읽고
2. `write(connfd, buf, 5)`로 다시 씁니다

결과:

```text
클라이언트 송신 5B
-> 서버 수신 5B
-> 서버 송신 5B
-> 클라이언트 재수신 5B
```

### 왜 Echo가 중요한가

웹 서버는 복잡해 보이지만, 본질은 Echo와 같습니다.

차이는:

- 웹 서버는 읽은 바이트를 HTTP 규칙으로 해석하고
- 파일을 읽거나 프로그램을 실행해
- 다시 바이트를 써 준다는 점입니다

### 이 절을 이해했다는 뜻

- 네트워크도 결국 read/write 관점으로 볼 수 있다는 감각이 생겼다면 성공입니다

## 11.5 Web Servers

### 한 줄 정의

웹 서버는 HTTP 규칙에 맞춰 웹 자원을 요청받고 응답하는 서버입니다.

### 왜 필요한가

11.4까지는 일반적인 네트워크 연결 API였습니다.

11.5부터는 그 연결 위에서 **무슨 프로토콜을 말하는지**를 다룹니다.

이번 장의 프로토콜은 `HTTP`입니다.

## 11.5.1 Web Basics

### 한 줄 정의

웹은 브라우저 같은 클라이언트가 URL로 자원을 식별하고, 웹 서버가 HTTP로 자원을 전달하는 시스템입니다.

### URL 분해

예시 URL:

```text
http://www.example.net:80/home.html
```

분해:

- scheme: `http`
- host: `www.example.net`
- port: `80`
- path: `/home.html`

Proxy Lab 스타일 URL:

```text
http://www.cmu.edu:8080/hub/index.html
```

분해:

- host: `www.cmu.edu`
- port: `8080`
- path: `/hub/index.html`

### 브라우저가 실제로 하는 일

초보자에게는 "브라우저가 HTML 한 개만 받는다"로 보일 수 있지만, 실제로는:

1. HTML을 받습니다
2. 그 안의 이미지, CSS, JS 참조를 파싱합니다
3. 추가 요청을 더 보냅니다

즉 웹은 **HTTP 트랜잭션의 연속**입니다.

### 초보자 오해 포인트

- URL의 host는 도메인 이름일 수 있고 IP일 수도 있습니다
- path는 서버 파일 시스템 절대경로와 항상 1:1 대응하지는 않습니다

## 11.5.2 Web Content

### 한 줄 정의

웹 컨텐츠는 크게 정적 컨텐츠와 동적 컨텐츠로 나뉩니다.

### static content

정적 컨텐츠는 서버에 저장된 파일을 거의 그대로 보내는 방식입니다.

예시:

- 파일: `home.html`
- 크기: `2048B`
- MIME type: `text/html`

### dynamic content

동적 컨텐츠는 요청에 따라 프로그램이 실행되어 그 결과를 만들어 내는 방식입니다.

예시:

- 프로그램: `adder`
- 입력: `15000`, `213`
- 결과: `15213`

### 정적과 동적의 차이를 한 줄로

- 정적: 이미 있는 바이트를 보낸다
- 동적: 지금 실행해서 만든 바이트를 보낸다

### 초보자 오해 포인트

- 정적이라고 해서 메모리를 안 쓰는 것은 아닙니다. 단지 내용이 미리 존재할 뿐입니다
- 동적이라고 해서 꼭 데이터베이스가 있어야 하는 것은 아닙니다. 간단한 계산 프로그램도 동적입니다

## 11.5.3 HTTP Transactions

### 한 줄 정의

HTTP 트랜잭션은 요청 메시지와 응답 메시지의 교환입니다.

### 요청 메시지 형식

우리 예시 요청:

```text
GET /home.html HTTP/1.0\r\n
Host: www.example.net\r\n
Connection: close\r\n
\r\n
```

구조:

1. 요청 줄
2. 헤더들
3. 빈 줄
4. 필요하면 바디

요청 줄:

```text
method URI version
```

우리 예시에서는:

- method: `GET`
- URI: `/home.html`
- version: `HTTP/1.0`

### 왜 빈 줄이 중요한가

HTTP 헤더는 텍스트 줄들의 묶음입니다.

따라서 서버는:

- 한 줄씩 읽다가
- `\r\n`만 있는 빈 줄을 만나면
- "헤더가 끝났다"라고 판단합니다

이 감각이 Tiny의 `read_requesthdrs`와 바로 연결됩니다.

### 응답 메시지 형식

예시:

```text
HTTP/1.0 200 OK\r\n
Server: Tiny Web Server\r\n
Content-length: 2048\r\n
Content-type: text/html\r\n
\r\n
[2048B body]
```

구조:

1. 상태 줄
2. 헤더들
3. 빈 줄
4. 바디

상태 줄:

```text
version status-code status-message
```

예시:

- version: `HTTP/1.0`
- status-code: `200`
- status-message: `OK`

### 상태 코드 감각

`CSAPP` 11장에서 자주 보게 되는 상태 코드:

- `200` -> 성공
- `301` -> 영구 이동
- `400` -> 잘못된 요청
- `403` -> 접근 금지
- `404` -> 파일 없음
- `501` -> 메서드 미구현
- `505` -> 지원하지 않는 HTTP 버전

### Content-length가 왜 중요한가

우리 정적 응답 예시:

- body 크기: `2048B`

따라서:

```text
Content-length: 2048
```

를 적습니다.

클라이언트는 이 값을 보고 바디를 얼마나 읽어야 할지 판단할 수 있습니다.

### Host 헤더가 왜 중요한가

같은 서버 IP 하나가 여러 사이트를 서비스할 수 있기 때문입니다.

즉:

- 접속 대상 IP/포트는 같아도
- `Host:` 값에 따라
- 어떤 사이트를 보여 줄지 달라질 수 있습니다

### GET, HEAD, POST 감각

11장의 중심은 `GET`입니다.

하지만 확장 관점에서는 아래 정도는 꼭 기억해 두는 것이 좋습니다.

- `GET`: 자원 요청, 쿼리가 URI에 붙는 경우가 많음
- `HEAD`: 바디 없이 헤더만 확인
- `POST`: 바디를 함께 보내는 요청

Tiny 기본 구현은 `GET` 중심이며, `HEAD`와 `POST`는 연습 문제 확장 포인트입니다.

### 초보자 오해 포인트

- HTTP는 텍스트 헤더와 바디가 섞인 구조이지, 무조건 HTML 한 덩어리만 받는 것이 아닙니다
- `Content-length`는 파일 시스템 크기와 완전히 같은 개념이 아니라 "이번 응답 바디 길이"입니다

### 이 절을 이해했다는 뜻

- 왜 `read_requesthdrs`가 빈 줄까지 읽어야 하는지 설명할 수 있습니다
- 왜 `501 Not Implemented`가 `GET` 외 메서드 처리와 연결되는지 설명할 수 있습니다

## 11.5.4 Serving Dynamic Content

### 한 줄 정의

동적 컨텐츠 서빙은 요청에 따라 프로그램을 실행하고, 그 프로그램이 만든 출력을 응답으로 보내는 방식입니다.

### CGI 예시

요청:

```text
GET /cgi-bin/adder?15000&213 HTTP/1.0
```

Tiny는 이것을 아래처럼 나눕니다.

- 프로그램 경로: `./cgi-bin/adder`
- query string: `15000&213`

### CGI에서 환경 변수로 넘기는 값

핵심 예시:

- `QUERY_STRING=15000&213`
- `REQUEST_METHOD=GET`
- `SERVER_PORT=15213`
- `REMOTE_ADDR=128.2.194.242`

만약 `POST`였다면:

- `CONTENT_LENGTH`
- `CONTENT_TYPE`

같은 값도 중요해집니다.

### 실제 숫자로 보는 동작

`adder` 프로그램이 query string을 읽고 아래처럼 파싱한다고 합시다.

```text
arg1 = 15000
arg2 = 213
sum  = 15213
```

그 결과로 HTML 또는 텍스트를 출력하면, 그 표준 출력이 그대로 클라이언트로 갑니다.

### 왜 stdout 리디렉션이 핵심인가

Tiny는 child process에서:

- `dup2(connfd, STDOUT_FILENO)`

를 수행해 CGI의 표준 출력을 소켓으로 연결합니다.

즉 CGI 입장에서는 그냥 `printf`했을 뿐인데, 그 결과가 네트워크를 타고 브라우저로 갑니다.

### 초보자 오해 포인트

- 동적 컨텐츠는 마법이 아니라 "프로그램 실행 결과를 응답으로 보낸다"는 뜻입니다
- CGI 프로그램은 직접 화면에 그리는 것이 아니라 바이트를 출력할 뿐입니다

## 11.6 Putting It Together: The Tiny Web Server

### 한 줄 정의

Tiny는 지금까지 나온 DNS 이후 연결, 소켓, HTTP, 정적/동적 컨텐츠 개념을 모두 합쳐 놓은 최소 웹 서버입니다.

### 이 절이 중요한 이유

11.6은 앞의 모든 조각을 "실제로 어떤 함수가 무슨 책임을 갖는가" 수준으로 결합합니다.

즉 11장의 학습 목표는 결국 Tiny를 읽고 설명할 수 있느냐로 수렴합니다.

## Tiny 전체 구조

Tiny의 큰 흐름은 아래와 같습니다.

```text
main
-> Accept
-> doit
   -> read_requesthdrs
   -> parse_uri
   -> serve_static or serve_dynamic
-> Close
```

이제 정적 요청과 동적 요청을 각각 숫자로 따라가겠습니다.

## Tiny 정적 요청 워크스루

### 1. main

```text
listenfd = Open_listenfd(15213)
while (1) {
  connfd = Accept(listenfd, ...)
  doit(connfd)
  Close(connfd)
}
```

숫자로 보면:

- 서버 포트: `15213`
- `listenfd = 3`
- 클라이언트가 들어오면 `connfd = 4`

즉:

- `3`은 계속 대기
- `4`는 이번 손님 전용 대화

### 2. Accept 직후 서버가 아는 정보

이번 손님 연결:

```text
(128.2.194.242:51213, 서버:15213)
```

이제 Tiny는 `connfd = 4`로 바이트를 읽을 수 있습니다.

### 3. doit가 요청 줄을 읽는다

첫 줄:

```text
GET /home.html HTTP/1.0\r\n
```

Tiny는 `sscanf` 수준으로 여기서:

- method = `GET`
- uri = `/home.html`
- version = `HTTP/1.0`

를 뽑습니다.

### 4. GET만 허용한다

만약 메서드가 `POST`나 `HEAD`면 기본 Tiny는:

```text
501 Not Implemented
```

를 돌려줍니다.

이건 "HTTP 전체를 다 구현하지 않고, 교육용으로 GET만 다룬다"는 의도입니다.

### 5. read_requesthdrs가 빈 줄까지 읽는다

예시 헤더:

1. `Host: www.example.net\r\n`
2. `Connection: close\r\n`
3. `Proxy-Connection: close\r\n`
4. `\r\n`

네 번째 줄이 빈 줄이므로 헤더 종료입니다.

### 6. parse_uri가 static인지 dynamic인지 판정한다

입력 URI:

```text
/home.html
```

출력:

- filename = `./home.html`
- cgiargs = `""`
- 판정 = static

`/`만 들어오면 보통:

```text
./home.html
```

같은 기본 파일을 붙입니다.

### 7. stat으로 파일 상태를 확인한다

가정:

- 파일 존재
- regular file
- 읽기 권한 있음
- 크기 `2048B`

그렇지 않으면:

- 없으면 `404`
- 권한이 없으면 `403`

같은 에러 응답을 만듭니다.

### 8. clienterror의 역할

에러가 난 경우 Tiny는:

- 상태 줄
- 헤더
- HTML 에러 바디

를 직접 만들어 보냅니다.

예를 들어 지원하지 않는 메서드면:

```text
HTTP/1.0 501 Not Implemented
```

상태를 보냅니다.

즉 Tiny는 "파일 보내는 함수"만 있는 게 아니라, **실패를 HTTP로 포장하는 함수**도 갖고 있습니다.

### 9. serve_static가 응답 헤더를 만든다

가정:

- 파일명: `./home.html`
- 크기: `2048B`
- 타입: `text/html`

헤더 예시:

```text
HTTP/1.0 200 OK\r\n
Server: Tiny Web Server\r\n
Content-length: 2048\r\n
Content-type: text/html\r\n
\r\n
```

헤더 크기 예시:

```text
91B
```

### 10. serve_static가 파일을 전송한다

흐름:

1. `open`으로 파일 열기
2. `mmap`으로 `2048B` 매핑
3. `rio_writen(connfd, srcp, 2048)`로 전송
4. `munmap`

전체 응답 전송량:

```text
헤더 91B + 바디 2048B = 2139B
```

### 11. 연결 종료

Tiny 기본 서버는 요청 하나 처리 후 연결을 닫는 단순한 모델로 생각하면 됩니다.

즉 순차 서버의 리듬은:

```text
accept -> handle one request -> close
```

입니다.

## Tiny 동적 요청 워크스루

이제 같은 Tiny가 CGI를 처리하는 경로를 봅니다.

### 1. 요청 줄

```text
GET /cgi-bin/adder?15000&213 HTTP/1.0\r\n
```

### 2. parse_uri 결과

- filename = `./cgi-bin/adder`
- cgiargs = `15000&213`
- 판정 = dynamic

### 3. stat으로 실행 가능 여부 확인

동적 컨텐츠는 regular file일 뿐 아니라 실행 권한도 중요합니다.

즉:

- regular file인가
- 실행 가능 비트가 있는가

를 확인해야 합니다.

### 4. serve_dynamic의 큰 흐름

Tiny는 대략 아래를 합니다.

1. 기본 응답 시작 줄을 보냅니다
2. `fork()`
3. child에서 `QUERY_STRING` 설정
4. child에서 `dup2(connfd, STDOUT_FILENO)`
5. child에서 `execve("./cgi-bin/adder", ...)`
6. parent에서 `wait`

### 5. 실제 숫자로 보면

질의 문자열:

```text
15000&213
```

CGI 내부 계산:

```text
15000 + 213 = 15213
```

child process는 결국 아래 성질을 가집니다.

- 입력 정보는 환경 변수에 들어 있고
- 출력은 소켓으로 나갑니다

즉 CGI 프로그램이 `printf`로 만든 바이트가 그대로 HTTP 응답 일부가 됩니다.

### 6. 왜 parent가 wait하는가

교육용 Tiny는 단순성을 위해 parent가 child 종료를 기다립니다.

이 방식은 이해하기는 쉽지만, 서버 전체를 잠깐 멈추게 만들 수 있습니다.

그래서 연습 문제에서는:

- `SIGCHLD`
- 비동기 reap

같은 확장이 나옵니다.

### 정적과 동적 차이를 다시 한 줄로

- 정적: 파일을 읽어 바로 보낸다
- 동적: 프로그램을 실행해 지금 만든 결과를 보낸다

## 11.7 Summary

### 한 줄 요약

11장을 끝까지 이해했다는 것은 아래 체인을 끊기지 않게 설명할 수 있다는 뜻입니다.

```text
도메인 이름
-> DNS로 IP 해석
-> IP + 포트로 socket address 형성
-> socket/connect 또는 socket/bind/listen/accept
-> 바이트 스트림 read/write
-> HTTP 요청/응답 해석
-> 정적 파일 또는 동적 프로그램 실행
-> Tiny 완성
```

### 11장을 마치며 남겨야 할 핵심 문장

- 네트워크는 결국 I/O다
- IP는 호스트를, 포트는 서비스를 구분한다
- 연결은 socket pair로 식별된다
- 소켓은 파일 디스크립터처럼 다룬다
- HTTP는 연결 위에서 오가는 텍스트 기반 규약이다
- Tiny는 이 모든 개념을 최소한으로 묶어 놓은 예제다

## Tiny -> Proxy -> SQL API 서버로 확장

### Tiny

- 요청을 직접 처리합니다
- 자원은 파일과 CGI입니다

구조:

```text
accept -> parse -> serve -> write back
```

### Proxy

- 요청을 받습니다
- 원 서버로 다시 연결합니다
- 요청과 응답을 중계합니다
- 필요하면 캐시합니다

즉 Tiny보다 한 단계 더 나가:

- 클라이언트 역할도 하고
- 서버 역할도 하는 프로그램입니다

### SQL API 서버

- 클라이언트 요청을 HTTP나 커스텀 프로토콜로 받습니다
- 내부 SQL 엔진으로 전달합니다
- 결과를 직렬화해 응답으로 보냅니다

자원만 바뀐 것입니다.

- Tiny의 자원 -> 파일
- Proxy의 자원 -> 원 서버 응답
- SQL API 서버의 자원 -> 테이블, 레코드, 인덱스, 실행 결과

### 수요코딩회 관점에서 재사용되는 것

여기서 그대로 재사용되는 개념:

- listening socket
- client connection fd
- request parsing
- worker에게 작업 분배
- response serialization
- 에러 응답 작성

즉 "네트워크를 모른 채 SQL API 서버를 짜는 것"은 매우 어렵고, 반대로 Tiny를 정확히 이해하면 서버 구현의 골격은 이미 잡힌 상태입니다.

## 이 문서가 완성되었는지 확인하는 체크리스트

아래를 막힘 없이 설명할 수 있으면, `CSAPP 11장`은 상당히 단단하게 이해한 것입니다.

- 왜 도메인 이름과 IP 주소를 분리하는가
- 왜 포트가 필요한가
- 왜 연결은 `(client_ip:client_port, server_ip:server_port)`로 식별되는가
- 왜 `listenfd`와 `connfd`를 나누는가
- 왜 HTTP 헤더는 빈 줄에서 끝나는가
- 왜 `Content-length`가 필요한가
- 왜 정적 컨텐츠와 동적 컨텐츠를 나누는가
- 왜 CGI는 `stdout`을 소켓으로 연결하는가
- 왜 Tiny는 학습용으로 좋고, 실서비스용으로는 부족한가
- 왜 Proxy와 SQL API 서버가 Tiny의 확장으로 보이는가

## 이 문서의 범위와 한계

이 문서는 `CSAPP 11장`을 초보자가 빠짐없이 이해하도록 돕는 문서입니다.

다만 아래 주제는 11장에서 깊게 다루지 않거나, 별도 학습이 더 필요합니다.

- TCP 3-way handshake 내부 디테일
- 혼잡 제어
- 재전송 알고리즘
- TLS / HTTPS
- HTTP/2, HTTP/3
- 멀티스레드/이벤트 기반 고성능 서버 설계

즉 이 문서를 다 이해하면:

- "네트워크 프로그래밍 입문"
- "Tiny/Proxy/Web 서버 구현 출발점"

으로는 충분히 강하지만, 네트워크 전체를 끝낸 것은 아닙니다.

그 대신 `CSAPP 11장` 자체는 아주 단단하게 아카이빙할 수 있는 수준을 목표로 했습니다.
